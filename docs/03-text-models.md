## 二、从宏观看 Transformer：数据怎么流动？

### 一张图总览

```
┌──────────────────────────────────────────────────────────┐
│  输入: "今天天气真"                                        │
└───────────────────────┬──────────────────────────────────┘
                        ▼
┌──────────────────────────────────────────────────────────┐
│  Step 1: Tokenize                                         │
│  "今天天气真" → [15432, 28901, 5678]                       │
│  查 tokenizer 词表，得到整数 ID                            │
└───────────────────────┬──────────────────────────────────┘
                        ▼
┌──────────────────────────────────────────────────────────┐
│  Step 2: Embedding                                        │
│  [15432] → [0.12, -0.34, ..., 0.55]   (2048 维向量)       │
│  [28901] → [0.78, 0.23, ..., -0.45]                       │
│  [5678]  → [0.11, 0.89, ..., 0.33]                        │
│  查 embedding 表，每个 token 变成向量                       │
└───────────────────────┬──────────────────────────────────┘
                        ▼
┌──────────────────────────────────────────────────────────┐
│  Step 3: Position Encoding                                │
│  给每个向量加上"我在第几个位置"的信息                        │
│  让模型知道 "今天" 在第 0 位，"天气" 在第 1 位              │
└───────────────────────┬──────────────────────────────────┘
                        ▼
┌──────────────────────────────────────────────────────────┐
│  Step 4: Transformer Block × N 层                         │
│  ┌────────────────────────────────────────────┐           │
│  │  RMSNorm          ← 归一化                  │           │
│  │      ↓                                      │           │
│  │  Self-Attention   ← token 之间互相"看"       │           │
│  │      ↓                                      │           │
│  │  Residual (+)     ← 残差连接               │           │
│  │      ↓                                      │           │
│  │  RMSNorm                                    │           │
│  │      ↓                                      │           │
│  │  Feed Forward     ← 每个 token 独立变换    │           │
│  │      ↓                                      │           │
│  │  Residual (+)                              │           │
│  └────────────────────────────────────────────┘           │
│     重复 28 层（Qwen2.5-1.5B 有 28 层）                    │
└───────────────────────┬──────────────────────────────────┘
                        ▼
┌──────────────────────────────────────────────────────────┐
│  Step 5: Output Projection (LM Head)                      │
│  最后一层的向量 → 151,936 维（词表大小）                    │
│  变成每个 token 的分数（logits）                           │
└───────────────────────┬──────────────────────────────────┘
                        ▼
┌──────────────────────────────────────────────────────────┐
│  Step 6: Softmax + Sampling                               │
│  logits → 概率分布 → 采样出下一个 token                    │
└──────────────────────────────────────────────────────────┘
```

### 一个关键事实

**Transformer 的 N 层是相同结构重复**，每层都对所有 token 的向量做一次变换：

```swift
// 宏观上就是这样一个循环
var hidden: [[Float]] = embeddingsWithPosition(tokens)  // [seq_len, 2048]

for layer in 0..<28 {  // Qwen2.5-1.5B 有 28 层
    hidden = transformerBlock(hidden, weights: layerWeights[layer])
}

let logits = outputProjection(hidden.last!)  // 只关心最后一个 token 的输出
let nextToken = sample(logits)
```

## 三、核心概念一：Self-Attention

这是 Transformer 最重要的机制，也是最容易讲玄学的部分。我们用**具体的数据**来讲。

### 3.1 要解决什么问题？

看这两个句子：

```
句子 1: "苹果掉到牛顿头上"        ← "苹果" = 水果
句子 2: "苹果发布了新 iPhone"      ← "苹果" = Apple 公司
```

同一个词"苹果"，在不同上下文里意思完全不同。模型怎么知道当前的"苹果"指什么？

**答案**：看它周围的词。Self-Attention 就是让每个 token 去"关注"其他 token，根据它们动态调整自己的含义。

### 3.2 用 iOS 类比：委托模式

```swift
// 每个 token 是一个 Cell
// 每个 Cell 想问"我在上下文里应该怎么理解自己？"
// 它去问其他所有 Cell，根据大家的回答加权平均

protocol TokenDelegate {
    func queryRelevance(to query: Token) -> Float  // 我跟你有多相关？
    func provideValue() -> [Float]                  // 你想知道的话，我给你这个信息
}

class Token: TokenDelegate {
    var embedding: [Float]
    
    func updateWithContext(_ others: [Token]) -> [Float] {
        // 1. 问所有其他 token："你跟我有多相关？"
        let relevanceScores = others.map { other in
            other.queryRelevance(to: self)
        }
        
        // 2. 归一化成权重（softmax）
        let weights = softmax(relevanceScores)
        
        // 3. 根据权重，把其他 token 的 value 加权求和
        var newEmbedding: [Float] = []
        for (other, weight) in zip(others, weights) {
            let value = other.provideValue()
            newEmbedding += value * weight
        }
        
        return newEmbedding  // 这个 token 融合了所有相关 token 的信息
    }
}
```

### 3.3 Q、K、V 到底是什么？

你会听到 Self-Attention 里的三个字母：Query、Key、Value。用 iOS 类比：

```
每个 token 身上有三份"简历":

Query (Q):  "我想找什么样的信息？"
           类比: NSFetchRequest 的查询条件

Key (K):    "我能提供什么样的信息？"
           类比: NSManagedObject 的索引字段

Value (V):  "如果你要我的信息，这是具体内容"
           类比: NSManagedObject 的实际属性值
```

**这三份"简历"是怎么来的？**

把 token 的原始 embedding 分别乘三个权重矩阵（W_q, W_k, W_v），得到三个新向量：

```swift
struct TokenAttentionView {
    let originalEmbedding: [Float]       // 2048 维
    
    // 通过学习得到的三个投影矩阵
    let W_q: [[Float]]  // [2048 × 128]
    let W_k: [[Float]]  // [2048 × 128]
    let W_v: [[Float]]  // [2048 × 128]
    
    // 计算 Q、K、V 向量（每个 128 维）
    var query: [Float] { matmul(originalEmbedding, W_q) }
    var key:   [Float] { matmul(originalEmbedding, W_k) }
    var value: [Float] { matmul(originalEmbedding, W_v) }
}
```

### 3.4 注意力打分：Query · Key

每个 token 用自己的 Query 跟所有其他 token 的 Key 做**点积**，得到相关性分数：

```swift
// 对于位置 i 的 token，计算它跟每个位置 j 的相关性
for i in 0..<seqLen {
    for j in 0..<seqLen {
        let score = dotProduct(token[i].query, token[j].key)
        // score 越大，表示 i 跟 j 越相关
    }
}

// 结果：一个 seq_len × seq_len 的相关性矩阵
```

### 3.5 归一化和加权

```swift
// 对每一行 softmax，得到归一化的"注意力权重"
for i in 0..<seqLen {
    let rawScores = (0..<seqLen).map { j in 
        dotProduct(token[i].query, token[j].key) / sqrt(128)  // 除以 sqrt(dim) 防数值爆炸
    }
    let weights = softmax(rawScores)  // 所有 j 的权重加起来 = 1
    
    // 加权求和所有 token 的 value，得到新的表示
    var newEmbedding = [Float](repeating: 0, count: 128)
    for j in 0..<seqLen {
        for d in 0..<128 {
            newEmbedding[d] += weights[j] * token[j].value[d]
        }
    }
    
    token[i].contextualEmbedding = newEmbedding
}
```

### 3.5.1 为什么要"加权求和 V"？这一步到底在做什么

这里是整个 Self-Attention 最容易卡住的地方。读者读到这里常有几个疑惑：

1. Q·K 已经算出相关性了，为什么不直接用？
2. V 到底扮演什么角色？
3. 加权求和 V 最终得到的是个什么东西？

把这三个问题彻底讲透，Self-Attention 就打通了。

#### 先搞清楚 Self-Attention 的终极产物

Self-Attention 的**最终输出**是一个新的 token 向量（跟输入形状一样），不是"相关性分数"。

```
输入:  100 个 token 向量    (每个 2048 维，独立无关联)
         ↓
       Self-Attention 层
         ↓
输出:  100 个新的 token 向量 (每个 2048 维，每个都融合了上下文信息)
```

它的任务是**"根据上下文，改写每个 token 的含义"**：

```
"苹果发布了新的iPhone"
    ↓
位置 0 的 "苹果" 的原始向量: [关于"水果"和"Apple公司"的含义混在一起]
    ↓ 经过 Self-Attention
位置 0 的 "苹果" 的新向量:   [已经偏向"Apple公司"的含义]
                              ↑ 因为上下文里有 "发布" 和 "iPhone"
```

#### 为什么不能直接用 Q·K 作为输出？

Q·K 的点积是一个**标量**（一个数字），只表示"有多相关"：

```
Q(苹果) · K(iPhone) = 4.9   ← 这只是个分数

但你要的是一个 2048 维的新向量，代表"苹果"的新含义！
仅凭一个数字造不出新向量，必须有别的东西提供"语义内容"。
```

**这就是 V 的作用：提供"实际语义内容"。**

#### Q、K、V 三者的角色分工

```
Query (Q):  "我在找什么样的上下文信息？"     → 负责"提问"
Key (K):    "我能被什么样的查询匹配到？"     → 负责"被匹配"
Value (V):  "如果被关注了，我实际提供什么？" → 负责"提供内容"

Q 和 K 负责"匹配打分"，V 负责"提供真实内容"。
```

**iOS 类比：搜索 API**

```swift
struct Document {
    let tags: [String]      // K: 搜索标签 ("Apple", "公司", "科技")
    let content: String     // V: 文档全文（真正的内容）
}

func search(query: String, documents: [Document]) -> String {
    // 1. 用 query 跟每个文档的 tags 匹配（Q·K）
    let scores = documents.map { matchScore(query: query, tags: $0.tags) }
    
    // 2. 归一化成权重（softmax）
    let weights = softmax(scores)
    
    // 3. 加权合并所有文档的 content（加权求和 V）
    //    像在做"多文档摘要"
    var result = ""
    for (doc, weight) in zip(documents, weights) {
        result += weight × doc.content   // ← 这就是"加权求和 V"
    }
    
    return result
}

// 注意:
//   tags（K） 只用来"匹配"
//   content（V） 才是你真正想要的"答案"
//   它们可以是完全不同维度的东西
```

#### K 和 V 为什么不能用同一个向量？

这是最精妙的设计。看一个例子：

```
位置 4 的 "iPhone" 这个 token：

  "iPhone" 的 K（索引标签）:
    "我是关于【科技产品、电子设备】的话题"
    ↑ 描述的是"在什么场景下我会被需要"
  
  "iPhone" 的 V（实际内容）:
    "我代表【苹果公司的智能手机】这个具体概念"
    ↑ 如果被关注了，我实际提供什么语义
```

K 像"简历上的技能标签"（用于匹配），V 像"实际能做的事情"（用于输出）。这两个**不必须完全对应**，所以要分开。

#### 完整例子：走一遍加权求和

```
句子: "苹果发布了新的 iPhone"
      位置 0: 苹果
      位置 1: 发布
      位置 2: 了
      位置 3: 新的
      位置 4: iPhone

现在关注位置 0 的 "苹果"，看它怎么更新自己的含义。

Step 1: "苹果" 的 Q 去和每个 token 的 K 做点积
─────────────────────────────────────────────
  Q(苹果) · K(苹果)    = 5.0  (自己和自己，最相关)
  Q(苹果) · K(发布)    = 4.8  (动作线索，很相关)
  Q(苹果) · K(了)      = 0.2  (虚词，不相关)
  Q(苹果) · K(新的)    = 1.5  (修饰语，一般相关)
  Q(苹果) · K(iPhone) = 4.9  (产品线索，很相关)

Step 2: Softmax 归一化成权重
─────────────────────────────────────────────
  w(苹果)    = 0.30
  w(发布)    = 0.28
  w(了)      = 0.02
  w(新的)    = 0.10
  w(iPhone) = 0.30

Step 3: 加权求和每个 token 的 V
─────────────────────────────────────────────
  V(苹果)   = [0.5, 0.2, ...]   ← "水果/Apple公司"混合含义
  V(发布)   = [0.1, 0.8, ...]   ← "动作/产品发布"含义
  V(了)     = [0.0, 0.0, ...]   ← 虚词，语义信息少
  V(新的)   = [0.3, 0.1, ...]   ← "新颖的"含义
  V(iPhone)= [0.9, 0.6, ...]   ← "智能手机/苹果产品"含义

  新的"苹果"向量 = 0.30 × V(苹果)
                 + 0.28 × V(发布)
                 + 0.02 × V(了)
                 + 0.10 × V(新的)
                 + 0.30 × V(iPhone)
                 
                = [0.4, 0.5, ...]
                  ↑
                  新向量明显偏向"科技公司/产品发布"
                  因为 "发布" 和 "iPhone" 的 V 权重很高
                  它们带来的"科技语义"被大量融入
                  而"水果"相关的语义没被强化
```

#### 这就是 Self-Attention 的魔法

```
"苹果" 原本向量:  含义模糊（水果 or 公司？）
                  ↓ Self-Attention
"苹果" 新向量:    含义明确（公司）

怎么做到的？
  ✓ 相关的 token（"发布"、"iPhone"）权重高
  ✓ 它们的 V 里包含"科技/产品"语义
  ✓ 被大量融入"苹果"的新向量
  ✓ 无关 token 的 V 几乎被忽略
```

#### 一句话记住

> **Q·K 决定"听谁的"，V 决定"听到什么内容"。**
> **加权求和 V = 根据相关性权重，把上下文中有用的语义"揉进"自己的新向量。**
>
> 所以每个 token 经过 Self-Attention 后，都变成了"原本的自己 + 相关上下文的语义融合"。

#### Swift 心智模型

```swift
class Token {
    // 我的三份"简历"
    var query: [Float]  // 我在找什么样的上下文
    var key:   [Float]  // 我能被什么样的查询匹配到
    var value: [Float]  // 我能提供什么具体的语义信息
    
    // Self-Attention = 我去"搜索"上下文，把搜到的语义融合进我自己
    func selfAttention(context: [Token]) -> [Float] {
        // 1. 用我的 Q 跟每个 token 的 K 做匹配，得相关性
        let scores = context.map { dotProduct(self.query, $0.key) }
        
        // 2. Softmax 归一化成权重（总和=1）
        let weights = softmax(scores)
        
        // 3. 按权重"摘要"出新向量：
        //    - 相关的 token，它们的 V 被大量采用
        //    - 不相关的 token，它们的 V 基本被忽略
        var newMe = [Float](repeating: 0, count: 128)
        for (token, weight) in zip(context, weights) {
            for i in 0..<128 {
                newMe[i] += weight * token.value[i]  // ← 关键！
            }
        }
        
        return newMe  // 我现在是"融合了上下文的新我"
    }
}
```

### 3.6 因果掩码（Causal Mask）：LLM 的关键约束

这里有一个**极其重要**的细节：生成式模型里，每个 token **只能看前面的 token，不能看后面的**。

```
生成 "我爱吃苹果":
  生成 "我" 时：只能看 <BOS>
  生成 "爱" 时：只能看 [<BOS>, 我]
  生成 "吃" 时：只能看 [<BOS>, 我, 爱]
  生成 "苹果" 时：只能看 [<BOS>, 我, 爱, 吃]
```

这条约束叫 **因果掩码（Causal Mask）**，它是所有生成式大模型（GPT/Llama/Qwen）的**根基**，没它整个架构就崩了。

#### 3.6.1 为什么需要因果掩码？—— 从训练方式说起

要理解因果掩码，**必须先理解 LLM 是怎么训练的**。

**LLM 的训练目标：预测下一个词**

LLM 不是"理解"了才回答，它在训练时只在做一件事：**给定前面的 token，预测下一个 token**。

```
训练数据: "我爱吃苹果"

模型学的是:
  看到 ""        → 应该预测 "我"
  看到 "我"      → 应该预测 "爱"
  看到 "我爱"    → 应该预测 "吃"
  看到 "我爱吃"  → 应该预测 "苹果"

这 4 个任务会同时训练（并行处理）
```

#### 3.6.2 关键矛盾：训练时如何"并行"做这些预测？

**天真的做法（极慢）**：

```
for 每条训练样本:
    for 每个位置 i:
        喂给模型: tokens[0..<i]
        让模型预测: tokens[i]
        计算损失

问题:
  一个 1000 字的句子要做 1000 次前向传播
  GPU 利用率极差
```

**聪明的做法（利用并行）**：

```
把整条句子一次喂给模型，同时对每个位置做预测:

输入:      [我, 爱, 吃, 苹果]
           ↓ 一次前向传播
输出:      [爱?, 吃?, 苹果?, EOS?]
             ↑    ↑    ↑     ↑
          pos 0 pos 1 pos 2 pos 3

4 个预测同时算，GPU 吃满 ✅
```

**但这里有致命问题**——如果不加掩码会怎样？

#### 3.6.3 如果没有因果掩码会怎样？（作弊问题）

在 Self-Attention 里，默认每个 token 可以看到所有 token（包括未来的）。这意味着：

```
训练时:
  位置 0 的 "我" 要预测下一个 token
  
  如果没有掩码:
    位置 0 的 "我" 可以看到位置 1 的 "爱"
    "爱" 已经告诉它答案了！
    它直接输出 "爱" 就行，零损失
  
  模型学到了什么？
    "预测下一个 token 时，直接抄下一个 token 就行"
```

**iOS 类比：让学生提前看答案**

```swift
// 没有因果掩码的训练 = 开卷考试，学生能看到答案
func train() {
    let sentence = "我爱吃苹果"
    
    for (i, token) in sentence.enumerated() {
        let prediction = model.predict(
            seeing: sentence,      // ← 看到完整句子！包括答案！
            at: i
        )
        // 学生作弊：直接从"能看到的答案"里抄
        // 损失 = 0，但它什么都没学到
    }
}
```

**实际效果：训练损失看起来很漂亮，但推理时瘫痪**：

```
无掩码训练（错误）:
  Epoch 1:  loss = 0.001  ← 完美？
  Epoch 2:  loss = 0.001
  
但推理时:
  用户输入: "我爱"
  模型: "我训练时总能看到答案，现在没答案可抄了..."
  输出: 乱码 / 胡言乱语 / 重复
  
因为推理时模型只能看到前面的 token，没有未来 token 可抄。
```

**这就是为什么必须有因果掩码**：让训练时"能看到什么"和推理时保持一致。

#### 3.6.4 因果掩码怎么实现？

**数学上：把未来位置的分数设为 -∞**

```
Self-Attention 的打分矩阵（4 个 token 的例子）:

原始分数（没有掩码）:
          K0    K1    K2    K3
  Q0 [  4.2,  3.8,  1.2,  0.5 ]
  Q1 [  2.1,  5.0,  4.3,  2.7 ]
  Q2 [  1.5,  3.9,  4.8,  3.2 ]
  Q3 [  0.8,  2.4,  3.6,  5.1 ]

加上因果掩码后:
          K0    K1    K2    K3
  Q0 [  4.2, -∞,  -∞,  -∞ ]    ← 位置 0 只能看 K0
  Q1 [  2.1,  5.0, -∞,  -∞ ]    ← 位置 1 只能看 K0, K1
  Q2 [  1.5,  3.9,  4.8, -∞ ]    ← 位置 2 只能看 K0-2
  Q3 [  0.8,  2.4,  3.6,  5.1 ]  ← 位置 3 能看全部

Softmax 后:
  Q0 [ 1.0,  0.0,  0.0,  0.0 ]    ← -∞ 经 softmax 变成 0
  Q1 [ 0.05, 0.95, 0.0,  0.0 ]
  Q2 [ 0.02, 0.33, 0.65, 0.0 ]
  Q3 [ 0.01, 0.06, 0.20, 0.73 ]

权重是 0 意味着:
  加权求和 V 时，未来 token 的 V 贡献为 0
  → 完全看不到未来的信息
```

**为什么用 -∞ 而不是直接设 0？**

```
如果直接把分数设成 0:
  softmax([4.2, 0, 0, 0]) = [0.66, 0.11, 0.11, 0.11]
  → 未来位置还是有 11% 的权重！没有真正屏蔽

设成 -∞:
  softmax([4.2, -∞, -∞, -∞]) = [1.0, 0.0, 0.0, 0.0]
  → 未来位置权重精确为 0，彻底屏蔽

巧妙利用了 softmax 的数学性质: e^(-∞) = 0
```

**Swift 实现**：

```swift
// 因果掩码
func applyCausalMask(scores: inout [[Float]]) {
    let seqLen = scores.count
    for i in 0..<seqLen {
        for j in 0..<seqLen {
            if j > i {
                scores[i][j] = -.infinity  // 位置 i 看不到未来的 j
            }
        }
    }
}

// 在 Self-Attention 中应用
func selfAttention(Q: [[Float]], K: [[Float]], V: [[Float]]) -> [[Float]] {
    var scores = matmul(Q, transpose(K))
    scores = scores / sqrt(Float(dimK))
    
    applyCausalMask(scores: &scores)  // ← 关键步骤！
    
    let weights = softmax(scores)
    return matmul(weights, V)
}
```

#### 3.6.5 因果掩码在哪一步使用？

**每一层 Self-Attention 内部，softmax 之前**。

```
完整的 Self-Attention 流程:
┌────────────────────────────────────────┐
│  输入: token 向量                        │
│    ↓                                    │
│  投影为 Q、K、V                          │
│    ↓                                    │
│  计算 scores = Q · K^T / √d             │
│    ↓                                    │
│  ⭐ 应用因果掩码 (causal mask)  ⭐        │  ← 这里！
│    把 j > i 的位置设为 -∞                │
│    ↓                                    │
│  weights = softmax(scores)              │
│    (-∞ 变成 0，未来 token 权重为 0)      │
│    ↓                                    │
│  output = weights · V                   │
│    ↓                                    │
│  输出: 融合了上下文的新向量               │
└────────────────────────────────────────┘

Qwen2.5-1.5B 有 28 层，每层都要应用因果掩码
```

**训练和推理时的区别**：

```
训练时:
  整句 "我爱吃苹果" 一次过模型
  每层 Self-Attention 应用因果掩码
  → 位置 0 看不到 "爱吃苹果"
  → 位置 1 看不到 "吃苹果"
  → 强制每个位置只能用前面的信息
  → 训练任务是"真的预测下一个"，不能作弊

推理时:
  已经生成 "我爱吃"，现在生成下一个
  位置 3 自然看不到位置 4（还不存在）
  因果掩码在推理时"自动满足"
  但代码里还是会应用掩码（统一逻辑）
```

#### 3.6.6 两种注意力方向对比

理解因果掩码最好的方式是对比"另一种"注意力：

**双向注意力（BERT 风格）——没有因果掩码**：

```
任务: 填空
输入: "我 [MASK] 吃苹果"

位置 1 的 [MASK] 需要看:
  - 左边: "我"      (告诉它主语)
  - 右边: "吃苹果"  (告诉它动作)
  → 双向都要看才能填对

BERT 完全不用因果掩码，训练任务也不是"预测下一个"
```

**单向注意力（GPT/Llama/Qwen）——必须有因果掩码**：

```
任务: 生成下一个 token
输入: "我 爱 吃"

位置 2 生成下一个 token 时:
  - 只能看 "我爱吃"
  - 如果能看到答案 "苹果"，那就是作弊

GPT 系列都是生成式的，必须用因果掩码
```

**对比表**：

| 特性 | BERT 风格 | GPT/Llama/Qwen 风格 |
|---|---|---|
| 注意力方向 | 双向 | 单向（只看过去） |
| 因果掩码 | ❌ 不用 | ✅ 必须用 |
| 训练任务 | 填空、分类 | 预测下一个 token |
| 用途 | 理解（NLU） | 生成（NLG） |
| 代表 | BERT, RoBERTa | GPT, Llama, Qwen |

#### 3.6.7 因果掩码如何让 KV Cache 成立？

这是一个隐藏的精妙设计，也是因果掩码**对客户端最重要的价值**。

```
因果掩码保证:
  "每个 token 只能看前面的 token"
  → token i 的 K/V 对 token i+1, i+2, ... 永远"可见"
  → 它们永远不会"消失"或"变化"
  
所以 KV Cache 能成立的前提就是因果掩码!

如果是双向注意力（BERT）:
  加入一个新 token 会改变所有 token 的 attention 权重
  → KV Cache 完全失效
  → 推理极慢

换句话说:
  "因果掩码" 是 "高效生成" 的根基
  没有它，大模型推理成本会翻百倍
```

**iOS 类比**：

```swift
// 因果掩码让每个 token 的 K/V 变成"不可变"
// 这和 Swift 的 let 一样的思想

struct Token {
    let position: Int
    let K: [Float]  // 一旦算出来就不变
    let V: [Float]  // 一旦算出来就不变
}

var kvCache: [Token] = []

func generateNext() -> Token {
    // 只看过去（因果掩码保证）
    // 所以 kvCache 中的 K/V 可以永久复用
    let newToken = computeWith(context: kvCache)
    kvCache.append(newToken)
    return newToken
}

// 如果是双向注意力:
//   每次加新 token，kvCache 里所有 K/V 都要重算
//   → 完全无法缓存
```

#### 3.6.8 一句话总结

> **因果掩码 = 强制模型"只能看过去"。它保证训练和推理的条件一致，让模型真正学会"预测"而不是"抄答案"。**
>
> **它是所有生成式大模型的基石，也是 KV Cache 能高效工作的前提。**

**对客户端的实际启示**：

```
为什么流式生成（token 一个个蹦）这么自然？
  → 因为因果掩码保证"已生成的 token 不会再变"
  → 可以放心地把每个 token 立刻显示给用户
  
为什么 KV Cache 这么有效？
  → 因为因果掩码保证"过去的 K/V 永远不变"
  → 一次计算，终身受益

所以因果掩码不只是训练技巧，它塑造了整个大模型的运行方式
和 iOS 上你看到的"打字机效果"息息相关
```

### 3.7 多头注意力（Multi-Head Attention）

单个 Q/K/V 只能学一种"关系"。实际模型会并行学很多种：

```
单头：
  一组 Q/K/V，学一种注意力模式
  比如: "关注语法关系"
  
多头（比如 16 头）：
  16 组独立的 Q/K/V，学 16 种不同的模式
  head 1: 关注"主谓关系"
  head 2: 关注"指代关系"（他/她/它指谁）
  head 3: 关注"修饰关系"（形容词修饰谁）
  ...
  head 16: 关注"长距离依赖"
```

但这里有个客户端开发者常问的问题：**多个头各自算完一个结果，最后怎么合并？是加权求和吗？哪个头说了算？**

答案可能跟你想的不一样——**不是加权求和，而是"拼接 + 投影"**。我们详细讲。

#### 3.7.1 多头的完整流程：从输入到输出

先看完整的数据流（以 Qwen2.5-1.5B 为例：d_model=1536，16 头，每头 96 维）：

```
输入: token 向量  [1536 维]
       │
       ▼
┌─────────────────────────────────────────────┐
│  Step 1: 投影并切分                          │
│                                              │
│  原始向量 1536 维                             │
│       ↓ W_q, W_k, W_v (三个大矩阵)           │
│  Q_all, K_all, V_all (都是 1536 维)           │
│       ↓ reshape 切成 16 份                    │
│  每个头拿到自己的 Q/K/V (96 维)               │
│                                              │
│  head 0:  Q_0 (96 维), K_0 (96 维), V_0 (96 维)│
│  head 1:  Q_1 (96 维), K_1 (96 维), V_1 (96 维)│
│  ...                                         │
│  head 15: Q_15(96 维), K_15(96 维), V_15(96 维)│
└─────────────────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────────────────┐
│  Step 2: 每个头独立做 Self-Attention         │
│                                              │
│  head 0:  独立算 attention → 输出 O_0 (96 维) │
│  head 1:  独立算 attention → 输出 O_1 (96 维) │
│  ...                                         │
│  head 15: 独立算 attention → 输出 O_15(96 维) │
│                                              │
│  16 个头之间完全不通信！完全独立！            │
└─────────────────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────────────────┐
│  Step 3: 拼接 (Concatenate)                   │
│                                              │
│  [O_0, O_1, ..., O_15]                       │
│   96  96         96                          │
│  ────────────────────────                    │
│   合并成 [1536 维]  ← 注意: 只是拼起来         │
│                      不是加权、不是求和        │
└─────────────────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────────────────┐
│  Step 4: Output Projection (W_o)              │
│                                              │
│  拼接结果 (1536 维)                           │
│       ↓ W_o 矩阵 [1536 × 1536]               │
│  最终输出 (1536 维)                           │
│                                              │
│  ⭐ W_o 的作用: 把 16 个头的结果"混合"一下     │
│     让不同头学到的信息互相交融                 │
└─────────────────────────────────────────────┘
       │
       ▼
输出: 融合多视角的 token 向量 [1536 维]
```

**关键点**：
- Step 2 是 16 个头**并行独立计算**，互不干扰
- Step 3 是**直接拼接**（concat），不是加权求和
- Step 4 才是让各头"交互"的地方——通过 W_o 投影矩阵

#### 3.7.2 "拼接"到底什么意思？为什么不是加权求和？

这是你最关心的问题。我用具体数字说清楚。

**拼接的本质：把 16 个短向量"首尾相连"变成一个长向量**

```
head 0 输出: [0.1, 0.2, 0.3, ..., 0.9]        96 个数字
head 1 输出: [0.5, 0.1, 0.4, ..., 0.2]        96 个数字
head 2 输出: [0.7, 0.8, 0.1, ..., 0.5]        96 个数字
...
head 15 输出: [0.3, 0.6, 0.2, ..., 0.8]       96 个数字

拼接结果:
  [0.1, 0.2, ..., 0.9, 0.5, 0.1, ..., 0.2, 0.7, ..., 0.8]
   └─ head 0 ─┘  └─ head 1 ─┘  └─ head 2 ─┘      └─ head 15 ─┘
  
总长度 = 16 × 96 = 1536 维
```

**iOS 类比：像 Array 的 append**

```swift
var combined: [Float] = []
for head in heads {
    combined.append(contentsOf: head.output)  // 依次拼上
}
// combined.count == 16 * 96 == 1536
```

#### 3.7.3 为什么不用加权求和？

你的直觉可能是："多个头各有意见，应该按权重投票决定"。但实际设计选择了拼接，原因如下：

**原因 1：加权求和会丢失信息**

```
加权求和 (假设权重都是 1/16):
  final = 1/16 × O_0 + 1/16 × O_1 + ... + 1/16 × O_15
  
  结果还是 96 维！
  → 16 个头辛辛苦苦学的不同视角，被平均成一碗粥
  → 等于只用了 1 个头的能力
  
拼接:
  final = [O_0, O_1, ..., O_15]
  
  结果是 1536 维！
  → 每个头学的信息完整保留
  → 后面的层能分别使用每个头的视角
```

**原因 2：加权求和假设"头之间可比较"，但它们学的根本不是同一个东西**

```
假设:
  head 0 关注"语法关系"，输出 [语法特征]
  head 1 关注"语义指代"，输出 [指代特征]

如果加权求和 = 0.5 × [语法特征] + 0.5 × [指代特征]
  → 这个操作没有意义！像是把苹果和橘子按重量混合
  → 最终向量变成了一个"四不像"

拼接 = [语法特征, 指代特征]
  → 各自保留，后面的层自己决定怎么用
```

**原因 3：用 W_o 投影来"学习如何组合"，比人为设定权重更聪明**

```
拼接后 (1536 维) → W_o 矩阵 → 最终输出 (1536 维)

W_o 是个学习出来的矩阵，它在训练中学会:
  "从 head 0 的维度，拿 0.3 的权重加到最终第 0 维"
  "从 head 1 的维度，拿 0.7 的权重加到最终第 0 维"
  "从 head 5 的维度，拿 -0.2 的权重加到最终第 100 维"
  ...
  
→ 每一个输出维度都有它独特的"从哪些头拿什么信息"的组合策略
→ 比简单的加权求和强大得多
```

#### 3.7.4 W_o 输出投影：多头的真正"融合"

W_o 是多头注意力最容易被忽视但至关重要的组件。

```
W_o 的形状: [d_model × d_model] = [1536 × 1536]

它做什么？
  拼接后的向量有 1536 维（16 个头各 96 维）
  W_o 把这 1536 维重新线性组合
  
  新维度 0 = w_00 × 原维度 0 + w_01 × 原维度 1 + ... + w_0_1535 × 原维度 1535
  新维度 1 = w_10 × 原维度 0 + w_11 × 原维度 1 + ... + w_1_1535 × 原维度 1535
  ...
  
  每一行 w_ij 决定: "新的第 i 维，从所有头的各个维度抽取多少权重"
```

**iOS 类比：调音台**

```swift
// 想象一个录音棚的调音台
// 16 个头 = 16 条音轨（各录了不同乐器）
// W_o = 调音师的操作面板

struct AudioMixer {
    let channels: [AudioChannel] = (0..<16).map { _ in AudioChannel() }  // 16 条音轨
    let mixingMatrix: [[Float]]  // W_o: 每个输出频道怎么混合 16 条音轨
    
    func mix() -> FinalTrack {
        // 1. 16 条音轨同时播放（独立）
        let allSamples = channels.flatMap { $0.samples }
        // 相当于 concat
        
        // 2. 混音台按 matrix 把它们组合成最终的立体声输出
        let final = matmul(allSamples, mixingMatrix)
        // 相当于 W_o 投影
        
        return final
    }
}
```

#### 3.7.5 完整的 Swift 心智模型

```swift
struct MultiHeadAttention {
    let numHeads = 16
    let dModel = 1536
    let headDim = 96   // = dModel / numHeads
    
    // 每个头有自己的 W_q, W_k, W_v
    let heads: [AttentionHead]
    
    // 最后的输出投影矩阵（共享）
    let W_o: [[Float]]  // [1536 × 1536]
    
    func forward(input: [Float]) -> [Float] {
        // Step 1-2: 每个头独立计算
        var headOutputs: [[Float]] = []
        for head in heads {
            let (Q, K, V) = head.project(input)          // 投影成 96 维
            let attended = head.selfAttention(Q, K, V)   // 独立 attention
            headOutputs.append(attended)                  // 96 维结果
        }
        
        // Step 3: 拼接（concat）
        // 不是加权求和！只是把 16 个 96 维向量首尾相连
        var concatenated: [Float] = []
        for output in headOutputs {
            concatenated.append(contentsOf: output)
        }
        // concatenated.count == 16 × 96 == 1536
        
        // Step 4: W_o 投影做"融合"
        let final = matmul(concatenated, W_o)
        return final  // 1536 维
    }
}
```

#### 3.7.6 从多头注意力到"预测下一个词"的完整链路

你的另一个问题：**算完多头之后，怎么预测下一个 token？**

多头注意力只是**一层** Transformer Block 的一部分。完整的链路是这样：

```
Token "苹果" 的 embedding [1536 维]
      │
      ▼
┌─── Transformer Block 1 ──────────┐
│  Multi-Head Attention (16 头)     │ ← 我们刚讲的这部分
│  输出: [1536 维]                  │
│      ↓ + 输入 (残差连接)          │
│  RMSNorm                         │
│      ↓                           │
│  Feed Forward (FFN)              │
│  输出: [1536 维]                  │
│      ↓ + 上一步 (残差连接)        │
└──────────────────────────────────┘
      │
      ▼
┌─── Transformer Block 2 ──────────┐
│  (同样结构)                       │
└──────────────────────────────────┘
      │
      ▼
    ... (共 28 层) ...
      │
      ▼
┌─── Transformer Block 28 ─────────┐
│  (最后一层)                       │
└──────────────────────────────────┘
      │
      ▼
最终 hidden state [1536 维]
      │
      ▼
┌─── Output Projection (LM Head) ──┐
│  W_out 矩阵 [1536 × 151936]      │
│  把 1536 维 → 151936 维 logits    │
└──────────────────────────────────┘
      │
      ▼
Logits [151936 维]
每一维对应词表里一个 token 的"分数"
      │
      ▼
Softmax → 概率分布
      │
      ▼
采样 (Temperature / Top-K / Top-P)
      │
      ▼
下一个 token 的 ID  ← 最终预测结果
      │
      ▼
查词表还原成文字: "公"  (比如生成"苹果公司"的"公")
```

**关键点梳理**：

1. **多头注意力的输出**是一个 1536 维向量（还不是"下一个词"）
2. 这个向量会**继续往下传**，经过 FFN，再进入下一层 Transformer Block
3. **重复 28 层**后得到最终的 hidden state（仍然是 1536 维）
4. 只有在**最后一层结束后**，才通过 LM Head（也叫 output projection）把 1536 维变成 151936 维（词表大小）
5. 151936 维的每一维是一个候选 token 的分数（logit）
6. Softmax 变成概率，采样出下一个 token

**所以多个头不直接决定"下一个词"。** 它们只是让每一层的 token 表示更丰富，最终由整个 28 层网络 + LM Head + 采样共同决定下一个 token。

#### 3.7.7 推理时只用"最后一个位置"的输出

这里有一个客户端同学容易忽视的细节。假设输入是 "今天天气真"（5 个 token），模型前向传播后会得到：

```
每个位置都有一个最终 hidden state:
  位置 0 "今"   → hidden_0 [1536 维]
  位置 1 "天"   → hidden_1 [1536 维]
  位置 2 "天"   → hidden_2 [1536 维]
  位置 3 "气"   → hidden_3 [1536 维]
  位置 4 "真"   → hidden_4 [1536 维]  ← 只用这个！

预测下一个 token 时，只用最后一个位置的 hidden state:
  logits = hidden_4 × W_out  [151936 维]
  
为什么？
  因为因果掩码保证: 位置 4 的 hidden_4 已经融合了前面所有 token
  位置 4 "代表"了整个输入序列的"最新状态"
  用它来预测下一个 token 最合适
```

**训练时有区别**：训练时会用**每个位置**的 hidden state 做预测（并行 teacher forcing），因为每个位置都是一个独立的"预测任务"（前面讲的因果掩码小节提到过）。

#### 3.7.8 回到你最初的问题

| 你的疑问 | 答案 |
|---|---|
| 16 个头的结果怎么合并？ | **拼接**（concat）成一个大向量，不是加权求和 |
| 是每个注意力有权重吗？ | 没有"头级别的权重"。真正的"权重"在 W_o 投影矩阵里，它学习如何从拼接结果提取信息 |
| 哪个头说了算？ | 没有"哪个头说了算"的概念。16 个头都贡献自己的信息，W_o 决定怎么组合 |
| 多头怎么决定下一个 token？ | 多头**不直接决定**。它只是让 token 表示更丰富，经过 28 层 + LM Head + 采样才决定下一个 token |
| 最后用谁的输出？ | 推理时只用**最后一个位置**的最终 hidden state 去预测下一个 token |

#### 3.7.9 一句话总结

> **多头注意力 = 16 个独立的"小 Attention"并行看数据的不同面 → 结果直接拼接 → 用 W_o 投影矩阵学习如何融合 → 当层输出**
>
> **多头不直接决定下一个 token。它只是 Transformer 中间的一个"特征提取模块"。最终的 token 预测，是 28 层 Transformer + LM Head + 采样器共同完成的。**

**iOS 类比收尾**：

```swift
// 多头注意力就像 UITableView 的多个 CellType
// - 每种 Cell 负责展示数据的一种视角
// - 它们独立渲染，不互相干扰
// - UITableView 的最终呈现 = 所有 Cell 的组合效果

// 但"最终要展示什么"不是由单个 Cell 决定的
// 而是由整个 Controller 的业务逻辑决定
// 多头注意力也是类似：它只是提供"多视角的特征"
// 最终决策由整个网络+采样器完成
```

### 3.8 GQA：iPhone 跑大模型的关键优化

现代端侧模型用的是 **Grouped Query Attention**，是多头注意力的一个优化版本。

```
标准多头 (MHA):
  16 个 Query 头、16 个 Key 头、16 个 Value 头
  → KV Cache 占用 × 16

GQA (分组):
  16 个 Query 头，但多个 Query 共享一组 K/V
  比如：16 个 Q 头，只有 4 个 K/V 头
  → KV Cache 占用 × 4  (减少 4 倍)

MQA (极端版):
  16 个 Q 头，但只有 1 组 K/V
  → KV Cache 占用 × 1  (减少 16 倍)
```

**为什么端侧特别重要？** 因为 KV Cache 是端侧推理的主要内存开销，对话越长越夸张：

```
Qwen2.5-1.5B 用 GQA（12Q/2KV）:
  每 token 的 KV Cache = 2 × 28层 × 2头 × 128dim × 2 bytes = 28.7 KB
  4096 token 对话 = 117 MB ✅

假如不用 GQA（标准 MHA 12Q/12KV）:
  每 token 的 KV Cache = 2 × 28层 × 12头 × 128dim × 2 bytes = 172 KB
  4096 token 对话 = 704 MB ❌ 直接 OOM
```

## 四、核心概念二：Feed Forward Network（FFN）

**FFN = Feed Forward Network（前馈神经网络）**，名字听起来高大上，但它是 Transformer 最容易被"带过"的部分。很多材料讲完 Attention 就跳过 FFN：

> "FFN 就是一个 MLP，没什么好说的。"

但实际上 **FFN 占了整个模型 60-70% 的参数**，它到底在干什么？我们必须搞清楚。

### 4.1 先回答"FFN 到底在做什么"

先看它跟 Attention 的分工：

```
┌─────────────────────────────────────────────────────────┐
│  Self-Attention 层                                        │
│                                                           │
│  干的事: token 之间"交流信息"                             │
│                                                           │
│  "苹果" ←→ "发布"                                         │
│  "苹果" ←→ "iPhone"                                       │
│   (所有 token 之间互相关注、融合上下文)                    │
│                                                           │
│  特点: 跨 token 的"水平"操作                              │
└─────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────┐
│  FFN 层                                                   │
│                                                           │
│  干的事: 每个 token 单独"深度思考 + 注入知识"              │
│                                                           │
│  "苹果融合了上下文" → 经过 FFN → "苹果的完整语义理解"    │
│                                                           │
│  特点: 单个 token 内部的"垂直"操作                        │
│        token 之间完全不交互                               │
│        引入非线性，让模型能学复杂变换                      │
└─────────────────────────────────────────────────────────┘
```

#### 用开会的类比记住 Attention 和 FFN 的区别

```
你在开会讨论一个问题:

Attention 层 = "开会讨论"
  大家互相交流观点，每个人把别人的意见融入自己的思考
  
FFN 层 = "个人整理笔记"
  会议结束后，每个人各自回座位
  消化刚才讨论的内容
  在自己的脑子里深度加工，形成自己的结论
  
一个 Transformer Block = "一轮会议 + 一轮整理"
  28 层 = 28 轮这样的迭代
  每轮下来，每个 token 的理解都更深一层
```

### 4.2 为什么必须有 FFN？两个核心原因

#### 原因 1：Attention 本质是"线性的"，没有 FFN 模型就没智能

这是最深刻的原因。

```
Attention 的计算本质:
  输出 = weighted_sum(V) = Σ w_i × V_i
  
  它只是在做"加权求和"
  输入的 V 是原向量的线性投影
  输出还是向量的线性组合

→ 整个 Attention 是"输入到输出的线性变换"
→ 没有非线性，模型能力非常有限
```

**类比：只会加减乘除，不会 if-else 的程序**

```swift
// 没有 FFN，模型只能做"线性运算"
func attention(input: [Float]) -> [Float] {
    // 所有操作都是加权求和、矩阵乘法
    // 数学上等价于一个大的线性变换
    return linearTransform(input)
}

// 有了 FFN，模型才能做"条件逻辑"
func ffn(input: [Float]) -> [Float] {
    var x = linear1(input)     // 线性变换
    x = activation(x)          // ⭐ 非线性！这里才是"判断/决策"
    x = linear2(x)
    return x
}
```

**神经网络的"智能"本质上来自非线性**。FFN 的激活函数（SwiGLU/GELU）提供了这个非线性。

#### 原因 2：FFN 是模型的"知识存储库"

2021 年 Google 的一项研究（"Transformer Feed-Forward Layers Are Key-Value Memories"）发现：

> **FFN 层实际上充当了一个巨大的"Key-Value 知识库"。**

```
FFN 的两层线性变换，可以被解读为:

Linear 1 (1536 → 8960): "Key 层"
  8960 个"关键模式检测器"
  每个检测器在问: "当前 token 是不是某种特定模式？"
  
  例如某些神经元可能检测:
    - "这是一个动物词吗？"
    - "这是一个编程相关的词吗？"
    - "这前面出现过否定词吗？"
    - "这是在描述颜色吗？"
    ...

Linear 2 (8960 → 1536): "Value 层"
  对应 8960 个"知识片段"
  如果某个 Key 被激活，它对应的 Value 就会加到输出上
  
  例如:
    - "动物检测器"激活 → 加入"动物相关语义"
    - "编程检测器"激活 → 加入"编程相关知识"
```

**这就是为什么 FFN 参数量这么大**（占 60-70%）——它在**存储模型学到的知识**。

### 4.3 具体例子：FFN 在处理"苹果"时在做什么

```
输入 Transformer Block 前:
  token "苹果" 的向量 [1536 维]
  (经过 Attention 已经融合了"发布"、"iPhone"等上下文)
  
现在进入 FFN 层:

Step 1: Linear 1 扩展到 8960 维
  把 1536 维的输入"展开"检测 8960 种模式
  
  神经元 #42  (检测"是否 Apple 公司"): 激活值 0.91  ✅ 强激活
  神经元 #128 (检测"是否水果"):        激活值 0.08  ❌ 弱激活
  神经元 #317 (检测"是否科技产品"):    激活值 0.85  ✅ 强激活
  神经元 #892 (检测"是否发布会语境"):  激活值 0.88  ✅ 强激活
  神经元 #1205(检测"是否食物"):        激活值 0.03  ❌ 弱激活
  ... (共 8960 个神经元)

Step 2: 激活函数（SwiGLU）
  把弱激活压得更小，强激活保持
  相当于"决策": 这些模式确实存在

Step 3: Linear 2 压回 1536 维
  根据哪些 Key 被激活，叠加对应的 Value（知识片段）
  
  "Apple 公司"激活 → 加入[关于 Apple 公司的语义向量]
  "科技产品"激活   → 加入[关于科技产品的语义向量]
  "发布会语境"激活 → 加入[关于产品发布的语义向量]
  
  最终输出 = 这些 Value 向量的叠加
  
  → "苹果"的向量被显著丰富
  → 从"模糊的苹果含义"变成"明确的 Apple 公司 + 发布语境 + 科技产品"
```

**对比 Attention 和 FFN 做的事**：

| 层 | 做的事 | "苹果"的变化 |
|---|---|---|
| Attention 之前 | 原始 embedding | 混合着水果和公司的含义 |
| **Attention 之后** | 融合了上下文 | 偏向"公司"，因为上下文有"发布"、"iPhone" |
| **FFN 之后** | 深度加工 + 注入知识 | 具备了"Apple 公司 + 科技 + 发布会"的丰富语义 |

### 4.4 FFN 的结构详解

理解了"做什么"之后，看具体结构就容易了。

```
每个 token 的 embedding（1536 维）
    ↓
Linear Layer 1 (W_up):  1536 → 8960    (扩大 ~6 倍，做"模式检测")
    ↓
激活函数（SwiGLU / GELU / ReLU）         ← ⭐ 非线性在这里
    ↓
Linear Layer 2 (W_down): 8960 → 1536   (缩回原尺寸，做"知识注入")
    ↓
输出（仍然是 1536 维）
```

现代模型用的是 **SwiGLU**，稍微复杂一点，多了一个"门控"矩阵：

```
SwiGLU 的计算:
  up   = input × W_up     (1536 → 8960)
  gate = input × W_gate   (1536 → 8960)
  activated = SiLU(gate) × up    ← 门控相乘，更灵活的非线性
  output = activated × W_down    (8960 → 1536)

SiLU(x) = x / (1 + e^(-x))    ← 一种平滑的激活函数
```

#### 为什么中间要扩展到 8960 维？

```
1536 维输入 → 8960 维中间 → 1536 维输出

为什么不直接 1536 → 1536？
  → 中间空间太小，"模式检测器"数量不够
  → 无法存储足够多的知识片段

为什么扩展到 8960 而不是 1 万？
  → 参数量和计算量的权衡
  → 6 倍是业界实验得出的甜点
```

### 4.5 FFN 的一个关键特性：每个 token 独立处理

这一点对理解性能和 MoE 很重要。

```
Attention 层:
  token 之间互相看、互相影响
  必须等所有 token 都到齐才能算
  
FFN 层:
  每个 token 独立处理，互不干扰
  100 个 token 可以开 100 个线程并行
  
代码上:
  for token in allTokens {
      token.newVec = ffn(token.vec)   // 互相不依赖
  }
```

**这带来了两个重要后果**：

**后果 1：FFN 天然适合并行**

```
Attention: token 之间有依赖，并行有限
FFN: 完全并行，GPU 吃满

这也是为什么 FFN 虽然参数多，但速度上不是瓶颈
```

**后果 2：这是 MoE 架构的基础**

```
MoE 把一个大 FFN 换成多个小专家 FFN + Router
为什么偏偏换 FFN？
  → 因为 FFN 是 token 独立处理的！
  → 可以给不同 token 分配不同专家
  → 如果换 Attention 会破坏跨 token 的交流
```

### 4.6 为什么 FFN 是参数大户？

```
Qwen2.5-1.5B 每层的参数量：
  
  Self-Attention 参数:
    W_q: 1536 × 1536 = 2.4M
    W_k: 1536 × 256  = 0.4M   (GQA，K 头少)
    W_v: 1536 × 256  = 0.4M
    W_o: 1536 × 1536 = 2.4M
    小计: 5.6M
  
  FFN 参数:
    W_up:   1536 × 8960 = 13.8M
    W_gate: 1536 × 8960 = 13.8M  (SwiGLU 的门控矩阵)
    W_down: 8960 × 1536 = 13.8M
    小计: 41.4M
  
  每层总计: 5.6 + 41.4 = 47M
  28 层总计: 1.3B  (再加上 embedding 和 LM head = 1.5B)
  
  结论: FFN 占了 ~88% 的参数！
```

这就是为什么 MoE 把 FFN 拆成多个专家而不是拆 Attention——**拆最大的那块才有意义**。

### 4.7 Swift 心智模型

```swift
// FFN 本质上是一个简单的类
class FeedForwardNetwork {
    let hiddenDim = 8960     // 中间扩展维度
    let modelDim = 1536      // 输入输出维度
    
    // 三个权重矩阵（SwiGLU）
    let W_up: [[Float]]      // [1536 × 8960] 扩展
    let W_gate: [[Float]]    // [1536 × 8960] 门控
    let W_down: [[Float]]    // [8960 × 1536] 压缩
    
    func forward(input: [Float]) -> [Float] {
        // 每个 token 独立处理（不跟其他 token 交互）
        
        // Step 1: 扩展维度并做门控（检测 8960 种"模式"）
        let up = matmul(input, W_up)        // [1, 8960]
        let gate = matmul(input, W_gate)    // [1, 8960]
        
        // Step 2: SwiGLU 激活（非线性决策）
        // 相当于"哪些模式被激活了"的判断
        var activated = [Float](repeating: 0, count: hiddenDim)
        for i in 0..<hiddenDim {
            let silu = gate[i] / (1 + exp(-gate[i]))  // SiLU 激活
            activated[i] = silu * up[i]                 // 门控乘以特征
        }
        
        // Step 3: 压缩回原维度（融合对应的"知识片段"）
        let output = matmul(activated, W_down)  // [1, 1536]
        
        return output
    }
}

// 完整的 Transformer Block
func transformerBlock(input: [[Float]]) -> [[Float]] {
    // 1. 多头注意力（跨 token 交流）
    var x = multiHeadAttention(input)
    x = add(x, input)  // 残差连接
    x = rmsNorm(x)
    
    // 2. FFN（每个 token 独立深度加工）
    var y = [[Float]]()
    for token in x {
        let processed = ffn.forward(input: token)  // 独立！
        y.append(processed)
    }
    y = add(y, x)     // 残差连接
    y = rmsNorm(y)
    
    return y
}
```

### 4.9 客户端视角：关于 FFN 的重要事实

```
1. FFN 是参数大户
   Qwen2.5-1.5B: FFN 占参数 ~88%
   → 量化 FFN 收益最大
   
2. 推理时的内存瓶颈
   FFN 的权重矩阵很大 (1536 × 8960 × 3 = 41M 参数/层)
   × 28 层 = 1.2B 参数
   → iPhone 上 FFN 的内存访问是主要开销

3. MoE 就是在改造 FFN
   把一个大 FFN 换成 8 个小专家 FFN + Router
   Attention 部分完全不动
   
4. GQA 和 FFN 无关
   GQA 是优化 Attention 的 K/V 头数
   跟 FFN 一点关系都没有
   所以即使用了 GQA，FFN 还是占大头

5. FFN 完全并行友好
   每个 token 独立处理
   Metal GPU 可以吃满
```

### 4.10 一句话总结

> **FFN = 每个 token 独立的"深度思考 + 知识注入"模块。**
>
> **Attention 让 token 之间交流信息，FFN 让每个 token 单独把信息"想透"并注入模型学到的知识。**
>
> **两者缺一不可：**
> **没有 Attention，模型不懂上下文；**
> **没有 FFN，模型是线性的、没有智能。**

## 五、残差连接与归一化：让深层网络训练成为可能

前面讲 Transformer Block 时反复出现一行代码：

```swift
x = add(x, input)  // ← 残差连接
```

这行"平平无奇"的加法，其实是 Transformer 另一个**至关重要**的设计。没有它，连 10 层以上的网络都训练不动，更别说 28 层、100 层。

### 5.1 残差连接是什么

**残差连接（Residual Connection，也叫 Skip Connection）= 让输入"跳过"中间层，直接加到输出上。**

```
没有残差连接的层:
  输入 x
    ↓
  层的计算 f(x)
    ↓
  输出 f(x)

有残差连接的层:
  输入 x
    ↓
  层的计算 f(x)       ┐
    ↓                │
  输出 = f(x) + x    │ ← 把原始输入"加回来"
                    ┘
```

就这么简单。**一个加法**。但它对训练深层网络至关重要。

### 5.2 在 Transformer 里长什么样

回顾前面多次出现的 Transformer Block 结构：

```swift
func transformerBlock(input: [[Float]]) -> [[Float]] {
    // 第一组：Attention + 残差
    var x = rmsNorm(input)
    x = multiHeadAttention(x)
    x = add(x, input)           // ← 残差连接 ①
    
    // 第二组：FFN + 残差
    var y = rmsNorm(x)
    y = ffn(y)
    y = add(y, x)                // ← 残差连接 ②
    
    return y
}
```

画成数据流图：

```
输入 x
  │
  ├──────────────────┐  (残差分支，原封不动)
  ▼                  │
RMSNorm              │
  ▼                  │
Multi-Head Attention │
  ▼                  │
  ●  + ●─────────────┘  ← ⭐ 残差连接①：加回原输入
  │
  ├──────────────────┐  (残差分支，原封不动)
  ▼                  │
RMSNorm              │
  ▼                  │
FFN                  │
  ▼                  │
  ●  + ●─────────────┘  ← ⭐ 残差连接②：加回上一步的输入
  │
  ▼
输出
```

**每个 Transformer Block 有两条残差连接**。Qwen2.5-1.5B 有 28 层，所以总共有 **56 条残差连接**。

### 5.3 为什么需要残差连接？—— 梯度消失问题

要理解残差连接，先理解它**解决了什么问题**。

#### 背景：深度 = 智能？但深了就训练不动了

```
直觉: 网络越深，能力越强
  1 层: 只能学简单关系
  10 层: 能学中等复杂度
  100 层: 应该能学很复杂的关系

但现实:
  1 层:  能训练
  10 层: 能训练
  30 层以上: 训练不动！损失下不去！
  
这是 2015 年之前深度学习最大的痛点。
```

#### 核心原因：梯度消失（Gradient Vanishing）

神经网络训练靠"反向传播"——把预测误差从输出层反向传回每一层，根据梯度调整参数。

```
误差信号的反向传递:
  输出层 ──→ 第 100 层 ──→ 第 99 层 ──→ ... ──→ 第 1 层
    梯度 1.0   0.5        0.25         ...     接近 0
  
每经过一层，梯度要乘以那层的"传导系数"
  如果系数 < 1，比如 0.5
  梯度 = 0.5^100 = 8 × 10^-31
  → 几乎为零！
  → 前面的层收不到误差信号
  → 无法学习
```

**iOS 类比**：就像给总部反馈问题，要逐层往上传。每传一层话就被"稀释"一点，到最高层已经听不清了，CEO 没法做正确决策。

### 5.4 残差连接怎么解决梯度消失？

**关键洞察**：残差连接给梯度开了一条"高速公路"。

#### 数学原理

```
没有残差连接:
  y = f(x)
  ∂y/∂x = ∂f/∂x     ← 梯度要经过 f 层传播，可能衰减

有残差连接:
  y = f(x) + x
  ∂y/∂x = ∂f/∂x + 1   ← ⭐ 多了一个 "+1"！
  
这个 "+1" 保证了无论 f 多复杂，梯度至少有 1 传回去
→ 梯度不会消失
```

#### 可视化

```
没有残差连接的深层网络（梯度传不回去）:

输出 ─梯度─ Layer28 ─梯度×0.5─ Layer27 ─ ... ─ Layer1
                                                  ↑
                                            梯度 ≈ 0
                                            这些层学不到东西

有残差连接的深层网络（梯度高速公路）:

       ┌──────高速公路 (梯度 = 1)───────────────┐
       │                                         │
       ▼                                         │
输出 ─梯度─ Layer28 ─梯度─ Layer27 ─ ... ─ Layer1
             │                                   │
             └──每层都有一条高速公路────────────┘
             
→ 梯度直接从输出传到最底层
→ 即使中间 f 很复杂也不影响
→ 能训练非常深的网络（几百层都没问题）
```

### 5.5 残差连接改变了网络的"学习目标"

这是更深一层的理解，也是"残差"这个名字的由来。

#### 没有残差：每层从零学完整映射

```
第 n 层的任务: 学一个完整的映射 f_n(x_n-1) = x_n
  每一层都要"完整地"重新学习
  非常困难
```

#### 有残差：每层只学"在上一层基础上的改进"

```
第 n 层的输出 = f_n(x_n-1) + x_n-1
             = (x_n-1 的改进) + x_n-1
             
→ 每一层只需要学"我要在上一层基础上改什么"
→ 如果本层"没啥贡献"，f_n 直接输出 0 就行，信息照样传下去
→ 任务变简单了

"残差" = "差异" = "改进量"
这就是名字的由来
```

#### iOS 开发者的类比

```swift
// 没有残差连接 = 每次迭代都从头重写整个文件
func noResidual(oldCode: String) -> String {
    return rewriteFromScratch()  // 每层从零开始，很累
}

// 有残差连接 = 每次迭代只加"diff"
func withResidual(oldCode: String) -> String {
    let changes = computeChanges(oldCode)  // 只学要改什么
    return oldCode + changes                // 在原基础上加改动
}

// 如果发现没啥要改的:
func withResidualNoChange(oldCode: String) -> String {
    let changes = ""              // 没改动
    return oldCode + changes       // 直接传下去
    // 比没有残差的情况下"强行重写一遍" 简单多了
}
```

### 5.6 实际效果：ResNet 的震撼 → Transformer 的基石

残差连接最早在 **ResNet (2015, He Kaiming 团队)** 里被提出，引起了深度学习界的轰动。

```
没有残差连接:
  VGG 网络 最深能到 ~20 层，再深就训练不动
  
有残差连接:
  ResNet-50:   50 层 ✅
  ResNet-101:  101 层 ✅
  ResNet-152:  152 层 ✅
  ResNet-1001: 1001 层 ✅（极端实验，证明可行）
  
一下子让"训练深度网络"从不可能变成了常规操作。
```

Transformer 继承了这个设计。现代大模型动辄几十上百层，全靠残差连接：

```
Qwen2.5-1.5B:  28 层,   56 条残差连接
Llama 3 8B:    32 层,   64 条残差连接
GPT-3:         96 层,  192 条残差连接
Llama 3.1 405B: 126 层, 252 条残差连接
```

### 5.7 加法的约束：形状必须一致

残差连接的实现非常简单，但有一个**硬性约束**：

```
y = f(x) + x

要求: f(x) 和 x 的形状必须完全一致
      (才能做加法)
```

**这就是为什么 Transformer 的每一层都保持 d_model 维度**（比如 1536）：

```
Transformer Block 内部维度流:
  输入:            [seq_len, 1536]   ← d_model
    ↓ Attention
  Attention 输出:  [seq_len, 1536]   ← 保持 1536 维
    ↓ + 输入
  残差后:          [seq_len, 1536]
    ↓ FFN
  FFN 输出:        [seq_len, 1536]   ← 即使中间扩到 8960，也要压回 1536
    ↓ + 上一步
  最终输出:        [seq_len, 1536]

→ 整个 Transformer 从头到尾，d_model 都保持不变
→ 这就是为什么 FFN 要"扩展再压缩"，而不是保持 8960
→ 为了能和输入做残差加法
```

### 5.8 归一化：残差连接的好搭档

残差连接会让数值"越加越大"（每层都加一次），所以通常会配合**归一化（Normalization）** 一起使用。

```
未归一化的问题:
  第 1 层输出: 值域 [-10, 10]
  残差加上:    [-20, 20]
  第 2 层输出: [-30, 30]
  ...
  第 28 层:    [-10000, 10000]  ← 数值爆炸！
  
归一化的作用:
  每经过一层都把数值"拉回"正常范围
  防止爆炸和消失
```

#### LayerNorm vs RMSNorm

现代大模型用的是 **RMSNorm**（Root Mean Square Normalization），是 LayerNorm 的简化版：

```
LayerNorm (原始 Transformer):
  1. 计算均值 μ 和方差 σ²
  2. 减均值，除方差: (x - μ) / σ
  3. 缩放加偏移: γ × (...) + β

RMSNorm (现代):
  1. 只计算 RMS: sqrt(mean(x²))
  2. 除 RMS: x / RMS
  3. 缩放: γ × (...)

优势:
  少了"减均值"和"加偏移"两步
  计算快约 10-20%
  效果几乎一样
  
所有现代开源大模型都用 RMSNorm:
  Llama 系列、Qwen 系列、Mistral、DeepSeek...
```

### 5.9 Pre-Norm vs Post-Norm：现代 Transformer 的微调

这是一个实现细节，但对训练稳定性很重要。

#### Post-Norm（原始 Transformer 2017）

```
y = LayerNorm(x + Attention(x))
```

先加残差，再归一化。**问题**：深层训练不稳定，需要仔细调学习率。

#### Pre-Norm（现代 GPT/Llama/Qwen）

```
y = x + Attention(LayerNorm(x))
```

先归一化再计算，最后加残差。**优势**：训练更稳定，适合深层模型。

```
所有现代大模型都用 Pre-Norm:
  Llama 系列: Pre-RMSNorm
  Qwen 系列:  Pre-RMSNorm
  GPT-2 之后: Pre-LayerNorm
```

这也是为什么前面的代码是：

```swift
// Pre-Norm 写法（现代）
var x = rmsNorm(input)              // 先归一化
x = multiHeadAttention(x)           // 再计算
x = add(x, input)                   // 最后加残差（注意加的是原输入）
```

### 5.10 一句话总结

> **残差连接 = 给每一层的输入开一条"高速公路"，直接加到输出上。**
>
> **它解决了深层网络的梯度消失问题，让训练几十上百层的 Transformer 成为可能。**
>
> **它还改变了学习目标——每一层只需要学"在上一层基础上怎么改进"，大大降低了学习难度。**

#### 对客户端的实际意义

```
1. 为什么大模型能做到几十层？
   → 靠残差连接，没它连 10 层都训练不动
   
2. 为什么 Transformer 每层维度都一样 (d_model)？
   → 残差连接的加法要求形状一致
   
3. 推理时有什么影响？
   → 几乎无影响，残差连接就是加法，极快
   → 但要求中间计算保留原输入，有一点点内存开销

4. 流式生成 / KV Cache 为什么能工作？
   → 残差连接让信息能从底层一路传到顶层
   → 每个 token 的信息在 28 层传播中不会丢失
```

## 六、位置编码：让模型知道 token 的顺序

前面讲了这么多 Transformer 的组件，但一个**看起来很基础**的问题一直没回答：

> **模型怎么知道 "我爱你" 和 "你爱我" 是不同的？**

答案是——**它本身不知道**。如果没有"位置编码"这个机制，对 Transformer 来说这两个句子是**完全一样**的。这一章就讲清楚这个隐藏的设计。

### 6.1 问题：Self-Attention 是"无序的"

先看一个反直觉的事实。

#### 回顾 Self-Attention 的计算

```
对于每个 token i:
  计算 Q_i
  计算它跟所有 token j 的分数 = Q_i · K_j
  softmax 得权重
  加权求和 V_j 得新向量
```

**关键观察**：这整个过程里，没有任何一步用到"token 的位置"。

```
句子 A: "我 爱 你"
  Token 0: 我    → 算 Q0, K0, V0
  Token 1: 爱    → 算 Q1, K1, V1
  Token 2: 你    → 算 Q2, K2, V2

句子 B: "你 爱 我"
  Token 0: 你    → 算 Q0 (还是和"你"在位置 2 时一样的 Q)
  Token 1: 爱    → 算 Q1
  Token 2: 我    → 算 Q2

Self-Attention 看到的:
  一个"集合" { 我, 爱, 你 }
  完全不知道它们的顺序！
  
→ 输出的向量组也完全一样
→ 模型无法区分这两个句子
```

#### iOS 类比：`Set` 不是 `Array`

```swift
// Self-Attention 本质上把 token 当成 Set 处理
let sentenceA: Set<String> = ["我", "爱", "你"]
let sentenceB: Set<String> = ["你", "爱", "我"]
print(sentenceA == sentenceB)   // true ❌ 这不对！

// 但自然语言显然是有序的 Array
let sentenceA_array = ["我", "爱", "你"]
let sentenceB_array = ["你", "爱", "我"]
print(sentenceA_array == sentenceB_array)   // false ✅
```

**所以 Transformer 必须额外告诉模型"这个 token 是第几个"。** 这就是位置编码的作用。

#### 为什么 Self-Attention 会设计成无序的？

这不是 bug，是特性。正因为它天生无序：

```
好处 1: 完全并行
  100 个 token 可以同时算
  如果有顺序依赖就只能串行（像 RNN 那样）
  → Transformer 比 RNN 快 10 倍以上

好处 2: 长距离依赖
  第 1000 个 token 可以直接看到第 1 个 token
  没有 RNN 那种"越传越弱"的问题
```

代价就是：**要手动把顺序信息"注入"进去**。

### 6.2 位置编码的本质：给每个位置一个"指纹"

**核心思路**：给每个位置 i 造一个独一无二的向量 `PE_i`，加到（或合并进）token 的 embedding 里。

```
普通 token embedding:
  "我" → [0.1, 0.2, 0.3, ...]   (跟位置无关)

加上位置编码后:
  位置 0 的 "我" → [0.1, 0.2, 0.3, ...] + PE_0 = [...]
  位置 5 的 "我" → [0.1, 0.2, 0.3, ...] + PE_5 = [...]
  
→ 同一个词在不同位置，最终向量不一样
→ 模型就能区分 "第 0 个我" 和 "第 5 个我"
```

**这就是位置编码的核心任务：给每个位置一个独特的"指纹"。**

### 6.3 方案一：Sinusoidal（原始 Transformer，2017）

最早的 Transformer 用正弦/余弦函数构造位置编码。

```swift
func sinusoidalPositionEncoding(position: Int, dim: Int) -> [Float] {
    var pe = [Float](repeating: 0, count: dim)
    for i in 0..<dim/2 {
        let angle = Float(position) / pow(10000, Float(2*i) / Float(dim))
        pe[2*i]   = sin(angle)    // 偶数维用 sin
        pe[2*i+1] = cos(angle)    // 奇数维用 cos
    }
    return pe
}
```

#### 效果可视化

```
每个位置得到一个 2048 维向量，像"位置指纹":

位置 0: [0.00, 1.00, 0.00, 1.00, 0.00, 1.00, ...]
位置 1: [0.84, 0.54, 0.01, 1.00, 0.00, 1.00, ...]
位置 2: [0.91, -0.42, 0.02, 1.00, 0.00, 1.00, ...]
...
位置 n: (独一无二的 2048 维向量)
```

#### 怎么用？

**加在 token embedding 上**（Element-wise 加法）：

```swift
// 输入 Transformer 之前:
for i in 0..<seqLen {
    let tokenEmb = embeddingTable[tokens[i]]           // 查词表得 token 向量
    let posEnc   = sinusoidalPositionEncoding(i, dim)  // 位置指纹
    
    finalInput[i] = tokenEmb + posEnc    // ← 加起来
}

// 然后把 finalInput 喂给 Transformer
```

#### 为什么是 sin/cos？

看似奇怪的选择，其实有数学含义：

```
sin/cos 的周期性 + 不同频率，让每个位置有独特组合
→ 而且满足: PE_(pos+k) 可以表示成 PE_pos 的线性函数
→ 意味着"相对位置"有简单的数学关系
→ 模型更容易学到"距离为 k 的 token 关系"
```

但 Sinusoidal 有个缺点——**它是"绝对位置"编码**，不直接表达"相对位置"。这就引出了 RoPE。

### 6.4 方案二：RoPE（旋转位置编码，现代首选）

所有现代开源大模型（Llama、Qwen、Mistral、DeepSeek）都用 **RoPE（Rotary Position Embedding）**。它跟 Sinusoidal 有两个关键区别：

1. **不是加法，而是旋转**
2. **不加在 embedding 上，而是加在 Q 和 K 上**

#### 核心思想：旋转 Q 和 K

```
Sinusoidal:  final_emb = token_emb + position_emb    (加法)
RoPE:        Q' = rotate(Q, position)                  (旋转 Q)
             K' = rotate(K, position)                  (旋转 K)
             然后用 Q' 和 K' 去算 attention
```

#### 可视化：向量在 2D 平面上旋转

```
位置 0 的 token:  向量不旋转 (角度 0°)
                  ●→
                  
位置 1 的 token:  向量顺时针旋转 θ
                  ●↗
                  
位置 2 的 token:  向量旋转 2θ
                  ●↑
                  
位置 3 的 token:  向量旋转 3θ  
                  ●↖
                  
...

每个位置对应一个独特的"旋转角度"
→ 相当于位置指纹
```

（实际上是把高维向量两两一组，每组在自己的 2D 平面上独立旋转。1536 维的向量 = 768 个 2D 平面，每个平面以不同频率旋转。）

#### RoPE 的魔法：相对位置天然浮现

这是 RoPE 最精妙的地方。看点积计算：

```
位置 5 的 Q（旋转 5θ） · 位置 3 的 K（旋转 3θ）
  
数学上可以证明:
  = 原点积 × cos((5-3)θ) + 原点积 × sin((5-3)θ)
  = 只跟 "5 - 3 = 2"（相对位置）有关！
  
→ 点积自动包含了相对位置的信息
→ 模型不用单独学"位置 5 和位置 3"
→ 只学"距离为 2 的 token 该怎么处理"
```

这就是 RoPE 的杀手级优势：**把绝对位置编码变成了相对位置编码，还不增加参数**。

#### RoPE 的优势

| 维度 | Sinusoidal | RoPE |
|---|---|---|
| 作用方式 | 加法（加在 embedding 上） | 旋转（应用在 Q/K 上） |
| 位置表达 | 绝对位置 | 相对位置 |
| 长度泛化 | 较差 | 优秀（能外推到训练没见过的长度） |
| 实现 | 预计算一个 PE 表 | 需要每层重新旋转 |
| 计算开销 | 低（加法） | 稍高（两次旋转）但仍很快 |
| **现代使用** | 几乎不用 | 所有主流大模型 |

### 6.5 ⭐ 位置编码用在哪个环节？（核心问题）

这个问题容易混淆，因为 Sinusoidal 和 RoPE 的答案**不同**。

#### Sinusoidal：用在 Transformer 之前，只用一次

```
Token IDs
    ↓
Embedding 查表        → 得到 token 向量
    ↓
+ 位置编码 (sinusoidal) → ⭐ 在这里加！只这一次
    ↓
输入到 Transformer Block 1
    ↓
Transformer Block 2
    ↓
... (后续层不再加位置编码)
    ↓
Transformer Block 28
    ↓
LM Head
```

#### RoPE：用在每一层的 Self-Attention 内部

```
Token IDs
    ↓
Embedding 查表        → 得到 token 向量 (不加位置编码)
    ↓
输入到 Transformer Block 1
    │
    ├─ Self-Attention:
    │    计算 Q, K, V
    │    → ⭐ 旋转 Q, K (应用 RoPE)   ← 在这里！
    │    → Q·K attention
    │    → 加权 V
    │
    ├─ FFN (不用位置编码)
    │
    ↓
Transformer Block 2
    │
    ├─ Self-Attention:
    │    → ⭐ 再次旋转 Q, K (应用 RoPE)   ← 每层都要！
    │
...
    ↓
Transformer Block 28
    │
    └─ ⭐ 最后一次应用 RoPE
```

**为什么 RoPE 要每层都做？**

```
因为 RoPE 只应用在 Q 和 K 上（不是 V、不是 embedding）
每一层都会重新计算 Q 和 K
所以每层都要重新"旋转"一次

这不是多余！每层的 Q/K 都是新的投影，需要重新注入位置信息
```

#### 代码对比

```swift
// Sinusoidal 版本
func forward(tokens: [Int]) -> [Float] {
    // Step 1: Embedding + 位置编码（只一次）
    var x = tokens.enumerated().map { (i, t) in
        embeddingTable[t] + sinusoidalPE(i)  // ← 在这里加
    }
    
    // Step 2: 过 28 层 Transformer（位置编码不再参与）
    for layer in 0..<28 {
        x = transformerBlock(x)
    }
    
    return x
}

// RoPE 版本
func forward(tokens: [Int]) -> [Float] {
    // Step 1: 纯 Embedding，不加位置编码
    var x = tokens.map { embeddingTable[$0] }
    
    // Step 2: 过 28 层 Transformer，每层内部应用 RoPE
    for layer in 0..<28 {
        x = transformerBlockWithRoPE(x)
    }
    
    return x
}

func transformerBlockWithRoPE(input: [[Float]]) -> [[Float]] {
    // ... (norm)
    var Q = matmul(input, W_q)
    var K = matmul(input, W_k)
    var V = matmul(input, W_v)
    
    // ⭐ 每层都要应用 RoPE
    Q = applyRoPE(Q, positions: [0, 1, 2, ..., seqLen - 1])
    K = applyRoPE(K, positions: [0, 1, 2, ..., seqLen - 1])
    // V 不用 RoPE！（因为 V 不参与 Q·K 的位置相关计算）
    
    let scores = matmul(Q, transpose(K)) / sqrt(d_k)
    // ... (mask + softmax + × V)
}
```

### 6.6 对客户端的重要影响

位置编码不只是"理论"，它直接影响端侧部署的几个关键指标：

#### 影响 1：上下文窗口的长度极限

```
模型训练时见过的最大长度 = 原生上下文窗口

Qwen2.5-1.5B 训练时: 32K tokens
Llama 3.2 3B 训练时: 128K tokens

问题：能不能让模型处理比训练时更长的序列？
  → Sinusoidal: 几乎不行，外推后效果崩
  → RoPE: 可以通过"调整旋转频率"扩展
```

#### 影响 2：RoPE 的 "theta" 参数调优

```
RoPE 有个超参数叫 "base theta"（通常是 10000）
控制了旋转频率

llama.cpp 可以调:

var ctxParams = llama_context_default_params()
ctxParams.n_ctx = 16384             // 希望用 16K 上下文
ctxParams.rope_freq_base = 10000    // 原始 theta
ctxParams.rope_freq_scale = 1.0     // 不缩放

// 如果模型训练只到 8K，想扩展到 16K:
ctxParams.n_ctx = 16384
ctxParams.rope_freq_scale = 0.5     // 频率减半 = 旋转慢一倍 = 覆盖更长距离
```

#### 影响 3：YaRN、PI 等长度扩展技术

这些是 RoPE 的扩展技巧，让训练 4K 的模型能跑 32K：

```
PI (Position Interpolation):
  把位置 "压缩"，让 16K 位置表现得像 8K
  简单但效果一般

YaRN (Yet another RoPE extensioN):
  更聪明的频率调整
  Qwen2.5、Llama 3.1 都用了

NTK-Aware Scaling:
  在不同频段用不同缩放比例
  效果比 PI 好
```

客户端用起来都是一个配置：`rope_freq_scale` 之类。

### 6.7 位置编码在整个流程的位置

把前面所有概念串起来，一张图看懂位置编码在大模型推理里的角色：

```
用户输入: "我爱你"
    ↓
Tokenize
    ↓
Token IDs: [15432, 28901, 5678]
    ↓
Embedding 查表
    ↓
Token 向量 (还没有位置信息)
    │
    ▼
╔════════════════════════════════════════╗
║  如果是 Sinusoidal: 这里加位置编码     ║
║  final_emb = token_emb + PE            ║
╚════════════════════════════════════════╝
    ↓
Transformer Block 1
  ├─ RMSNorm
  ├─ Self-Attention:
  │   ├─ 计算 Q, K, V
  │   ├─ ⭐ 如果是 RoPE: 在这里旋转 Q, K
  │   ├─ 应用因果掩码
  │   ├─ Softmax(Q·K / √d)
  │   └─ 加权求和 V
  ├─ 残差连接
  ├─ RMSNorm
  ├─ FFN
  └─ 残差连接
    ↓

  └─ 同上 (RoPE 每层都做)
    ↓
... 共 28 层 ...
    ↓

    ↓

    ↓
采样下一个 token
```

### 6.8 一句话总结

> **位置编码 = 给每个位置一个独一无二的"指纹"，让 Transformer 知道 token 的顺序。**
>
> **没有它，"我爱你" 和 "你爱我" 对模型来说一模一样——因为 Self-Attention 本身是无序的。**

#### Sinusoidal vs RoPE 速查

| | Sinusoidal | RoPE |
|---|---|---|
| 作用位置 | Embedding 之后，Transformer 之前（只一次） | 每层 Self-Attention 内部，应用在 Q/K |
| 表达方式 | 加法 | 旋转 |
| 位置语义 | 绝对位置 | 相对位置 |
| 长度外推 | 差 | 好 |
| 现代使用 | 已弃用 | 主流（Llama、Qwen、Mistral 等全部使用） |

#### 客户端的实际启示

```
1. 看模型 config 里的 rope_theta
   → 决定了位置编码的频率，影响长度泛化
   
2. 看 n_ctx（上下文窗口）
   → 想扩展超过模型原生长度，用 rope_freq_scale
   
3. RoPE 只改 Q 和 K，不改 V 和 embedding
   → V 和 embedding 跟位置无关，只表达"内容"
   
4. 为什么 LLM 能处理超长上下文？
   → 靠 RoPE + YaRN/PI 等扩展技巧
   → 没 RoPE 的老模型做不到
```

## 七、Chat Template：被大家忽视但极其重要的"格式"

### 6.1 为什么 Chat Template 是大坑？

很多新手同学把模型当成"输入文本 → 输出文本"的黑盒。但其实模型对**输入的格式**极其敏感。

```
❌ 错误用法:
  直接输入: "你好"
  模型输出: "你好...你好...你好..." (胡言乱语)

✅ 正确用法（Qwen2.5 的 chat template）:
  输入: "<|im_start|>system\nYou are a helpful assistant.<|im_end|>
        <|im_start|>user\n你好<|im_end|>
        <|im_start|>assistant\n"
  模型输出: "你好！有什么我可以帮你的？"
```

**为什么？** 因为模型在训练时，对话数据都是按这个格式标注的。输入格式不对，模型根本不知道现在该生成什么。

### 6.2 常见模型的 Chat Template

```
Qwen2.5:
  <|im_start|>system
  {system_message}<|im_end|>
  <|im_start|>user
  {user_message}<|im_end|>
  <|im_start|>assistant
  
Llama 3.x:
  <|begin_of_text|><|start_header_id|>system<|end_header_id|>
  {system_message}<|eot_id|><|start_header_id|>user<|end_header_id|>
  {user_message}<|eot_id|><|start_header_id|>assistant<|end_header_id|>
  
Gemma 2:
  <start_of_turn>user
  {user_message}<end_of_turn>
  <start_of_turn>model
  
Phi-3:
  <|system|>
  {system_message}<|end|>
  <|user|>
  {user_message}<|end|>
  <|assistant|>
```


### 6.4 特殊 Token（Special Token）

上面那些奇怪的 `<|im_start|>` 是**单一 token**，在词表里占一个 ID，不是普通字符串。

```
<|im_start|>  → token_id: 151644
<|im_end|>    → token_id: 151645
<|endoftext|> → token_id: 151643  (遇到就停止生成)
```

客户端拿到输出时，要**过滤掉**这些特殊 token 再展示给用户。

## 八、量化：让大模型塞进 iPhone

### 8.1 什么是量化？

模型训练完后参数都是 FP16（16 位浮点），太占地方。量化就是**用更少的 bit 表示同一个数**。

```
原始 FP16: 0.123456 (16 bit, 2 字节)
         ↓
量化 INT8: 12       (8 bit,  1 字节)   ← 还得配一个缩放因子 scale=0.01
         ↓ 反量化时
恢复:     12 × 0.01 = 0.12   (有误差但可接受)
```

> **补充说明**：上面这个例子描述的"用 8-bit 整数 + 一个 scale 缩放因子"的方式，就是 **Q8_0** 风格的量化。后面我们会看到，更现代的 K 系列量化在这个基础上做了更精细的优化。

### 8.2 量化为什么可行？

因为神经网络有**冗余**和**容错性**：

1. 很多参数非常接近 0，可以用小数表示
2. 小扰动对最终输出影响不大（梯度平滑）
3. 重要参数可以单独保留高精度

### 8.3 用具体数字看每种量化

抽象的名字不如具体数字直观。假设我们有这样一组真实的模型权重：

```
原始 FP16 权重 (4 个数):
  [0.123, -0.087, 0.456, -0.032]
  
每个数占 16 bit = 2 字节
4 个数 = 8 字节
```

下面用同一组权重，走一遍每种量化方案。

#### 8.3.1 Q8_0：最简单的 8-bit 量化

```
步骤 1: 找出最大绝对值
  max(|0.123|, |-0.087|, |0.456|, |-0.032|) = 0.456

步骤 2: 计算 scale
  scale = 0.456 / 127  ≈ 0.0036
  (INT8 的范围是 -128 ~ 127，所以用 127 作为分母)

步骤 3: 量化每个值
  0.123  → round(0.123 / 0.0036)  =  34
  -0.087 → round(-0.087 / 0.0036) = -24
  0.456  → round(0.456 / 0.0036)  = 127
  -0.032 → round(-0.032 / 0.0036) =  -9

存储:
  scale (16 bit) + 4 个 INT8 (32 bit) = 48 bit = 6 字节
  
  对比原来 8 字节 → 节省 25%（单独看 4 个数）
  实际上 Q8_0 按 32 个数一组，真实节省约 50%

反量化 (推理时):
  34  × 0.0036 =  0.122  (原 0.123)
  -24 × 0.0036 = -0.087
  127 × 0.0036 =  0.457  (原 0.456)
  -9  × 0.0036 = -0.032
  
  → 几乎无损！这就是"几乎无损"的由来
```

**Q8_0 的特点**：一组 32 个数共享一个 scale，简单直接。

#### 8.3.2 Q4_0：最简单的 4-bit 量化

```
同样的 4 个数: [0.123, -0.087, 0.456, -0.032]

步骤 1: 找最大绝对值 = 0.456

步骤 2: 计算 scale (4-bit 范围 -8 ~ 7)
  scale = 0.456 / 7 ≈ 0.0651

步骤 3: 量化
  0.123  → round(0.123 / 0.0651)  =  2    (4-bit: 0010)
  -0.087 → round(-0.087 / 0.0651) = -1    (4-bit: 1111)
  0.456  → round(0.456 / 0.0651)  =  7    (4-bit: 0111)
  -0.032 → round(-0.032 / 0.0651) =  0    (4-bit: 0000)

存储:
  scale (16 bit) + 4 个 4-bit = 16 + 16 = 32 bit = 4 字节
  
反量化:
  2  × 0.0651 =  0.130  (原 0.123, 误差 0.007)
  -1 × 0.0651 = -0.065  (原 -0.087, 误差 0.022)  ← 误差变大！
  7  × 0.0651 =  0.456
  0  × 0.0651 =  0
  
→ 误差比 Q8_0 大很多
→ 这是"精度换空间"的本质
```

**问题**：Q4_0 精度明显下降，尤其是数值接近 0 的时候。这就是为什么需要更聪明的方案。

#### 8.3.3 Q4_1：带偏移量（稍微好一点）

```
Q4_0 的假设: 数据分布是"对称"的 (-max 到 +max)
Q4_1 的改进: 假设数据可能"偏"一侧，加一个偏移量

公式:
  quantized = round((original - min) / scale)
  recovered = quantized × scale + min

更准但多存了一个 min 值:
  Q4_0: scale (16 bit)
  Q4_1: scale (16 bit) + min (16 bit)  ← 多占一点空间
  
提升: 稍微更准，但文件大一点
```

**这就是你看到的"Q4_1 带偏移量"。**

#### 8.3.4 Q8_0 vs Q4_0 对比

用更直观的图展示：

```
FP16 原始值 (x 轴) → 量化后的离散值 (y 轴)

Q8_0 (8-bit, 256 个档位):
   ●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●
   ●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●
   ●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●●
   档位密集，能精确表示多种值
   
Q4_0 (4-bit, 16 个档位):
   ●    ●    ●    ●    ●    ●    ●    ●
   ●    ●    ●    ●    ●    ●    ●    ●
   只有 16 个档位，误差明显
   所有 FP16 值都要"舍入"到最近的档位
```

### 8.4 K 系列：现代端侧量化的核心

Q4_0 和 Q4_1 有一个**根本缺陷**：**整组数据共享同一个 scale**。

```
问题场景:
  一组 32 个数里:
    大部分数: [0.001, 0.002, 0.003, ...]  (小)
    但有 1 个异常值: 5.0                    (大)
  
  如果用 Q4_0 (一组共享 scale):
    scale = 5.0 / 7 ≈ 0.714
    0.001 → round(0.001 / 0.714) = 0
    0.002 → round(0.002 / 0.714) = 0
    0.003 → round(0.003 / 0.714) = 0
    ...
    → 所有小数都变成 0！全部信息丢失 ❌
```

**K 系列的核心改进**：**更细粒度的分组 + 每组独立 scale**。

#### 8.4.1 Q4_K 的真实结构

```
Q4_K 的分组方式:

  "超级块" (super-block): 256 个权重
     ↓ 分成
  "子块" (sub-block): 32 个权重，共 8 个子块
  
存储布局:
  超级块开头:
    - 1 个 FP16 scale  (给 8 个子块的 scale 用的再缩放)
    - 1 个 FP16 min    (给 8 个子块的 min 用的再缩放)
  
  每个子块:
    - 6-bit 的子块 scale    (压缩后的 scale)
    - 6-bit 的子块 min      (压缩后的 min)
    - 32 个 4-bit 的权重值
  
结果:
  每个子块自己有 scale 和 min
  不会因为一组里有异常值导致整组精度崩溃
```

**iOS 类比**：

```swift
// Q4_0 像给所有员工发同一个"通用工具"
struct Q4_0_Block {
    let sharedScale: Float   // 整组共用
    let values: [UInt4]      // 32 个 4-bit 值
}

// Q4_K 像给每个小组发"专属工具"
struct Q4_K_SubBlock {
    let scale: UInt6         // 每个子块独立的 scale
    let min: UInt6           // 每个子块独立的 min
    let values: [UInt4]      // 32 个 4-bit 值
}

struct Q4_K_SuperBlock {
    let globalScale: Float   // 给所有 scale 再缩放
    let globalMin: Float     // 给所有 min 再缩放
    let subBlocks: [Q4_K_SubBlock]  // 8 个子块
}
```

#### 8.4.2 K 系列的"混合精度"特性

这是 K 系列**最聪明**的地方：不同张量用不同量化精度。

```
Q4_K_M 的实际内部:
  
  Attention 的 Q/K 权重:  用 Q6_K 量化 (更重要，精度要求高)
  Attention 的 V 权重:    用 Q4_K 量化
  Attention output:       用 Q6_K 量化
  FFN gate/up:            用 Q4_K 量化
  FFN down:               用 Q6_K 量化 (输出层更敏感)
  LM Head:                用 Q6_K 量化
  
→ 名字叫 "Q4_K_M"，但内部其实是 Q4_K 和 Q6_K 的混合
→ M = Medium，意思是"中等"，用更多 Q6_K
```

#### 8.4.3 S / M / L 的区别

```
Q4_K_S (Small):   更多层用 Q4_K，少用 Q6_K    → 最小 / 质量稍差
Q4_K_M (Medium):  重要层用 Q6_K               → 平衡 (推荐!)
Q4_K_L (Large):   更多层用 Q6_K 或 Q8         → 最大 / 质量最好

举例 (Qwen2.5-1.5B):
  Q4_K_S: ~900 MB
  Q4_K_M: ~1.0 GB
  Q4_K_L: ~1.1 GB
  
大小差 10% 左右，质量差一点，根据 iPhone 内存选择。
```

#### 8.4.4 用具体数字走一遍 Q4_K 的转换

前面讲了原理和结构，但还不够直观。我们用一个"玩具例子"完整走一遍：**一组权重怎么被 Q4_K 压缩，怎么恢复回来**。

**造一组模拟数据**

为了简化，用 **12 个权重**（分 3 个子块，每子块 4 个）代替真实的 256 个：

```
子块 0: [0.12, -0.08, 0.15, -0.11]        (范围 -0.11 ~ 0.15，小)
子块 1: [0.44, -0.31, 0.55, -0.45]        (范围 -0.45 ~ 0.55，大)
子块 2: [0.05, -0.03, 0.08, -0.02]        (范围 -0.03 ~ 0.08，非常小)
```

**关键观察**：不同子块的数值范围差很多。这正是 Q4_K 能"打赢" Q4_0 的场景。

**Step 1：每个子块独立统计最大、最小值**

```
子块 0: min=-0.11, max=0.15
子块 1: min=-0.45, max=0.55
子块 2: min=-0.03, max=0.08
```

**Step 2：每个子块独立计算自己的 scale**

```
4-bit 无符号范围是 0~15，所以:
  sub_scale = (max - min) / 15

子块 0: scale = (0.15 - (-0.11)) / 15 = 0.26 / 15 ≈ 0.01733
子块 1: scale = (0.55 - (-0.45)) / 15 = 1.00 / 15 ≈ 0.06667   ← 大范围用大 scale
子块 2: scale = (0.08 - (-0.03)) / 15 = 0.11 / 15 ≈ 0.00733   ← 小范围用小 scale
```

**子块 2 的 scale 极小（0.00733）是关键**，这让它能精确表达 `0.03` 这种小数。

**Step 3：用各自的 scale 量化（量化公式：`q = round((w - min) / scale)`）**

```
子块 0 (scale=0.01733, min=-0.11):
  0.12  → round((0.12 - (-0.11)) / 0.01733)  = round(13.27) = 13
  -0.08 → round((-0.08 - (-0.11)) / 0.01733) = round(1.73)  = 2
  0.15  → round((0.15 - (-0.11)) / 0.01733)  = round(15.00) = 15
  -0.11 → round((-0.11 - (-0.11)) / 0.01733) = round(0.00)  = 0
  存储: [13, 2, 15, 0]  (每个用 4 bit)

子块 1 (scale=0.06667, min=-0.45):
  0.44  → round((0.44 - (-0.45)) / 0.06667)  = round(13.35) = 13
  -0.31 → round((-0.31 - (-0.45)) / 0.06667) = round(2.10)  = 2
  0.55  → round((0.55 - (-0.45)) / 0.06667)  = round(15.00) = 15
  -0.45 → round((-0.45 - (-0.45)) / 0.06667) = round(0.00)  = 0
  存储: [13, 2, 15, 0]

子块 2 (scale=0.00733, min=-0.03):
  0.05  → round((0.05 - (-0.03)) / 0.00733)  = round(10.91) = 11
  -0.03 → round((-0.03 - (-0.03)) / 0.00733) = round(0.00)  = 0
  0.08  → round((0.08 - (-0.03)) / 0.00733)  = round(15.00) = 15
  -0.02 → round((-0.02 - (-0.03)) / 0.00733) = round(1.36)  = 1
  存储: [11, 0, 15, 1]
```

**Step 4：反量化验证（`w = q × scale + min`）**

```
子块 0 反量化:
  13 × 0.01733 + (-0.11) = 0.1153  (原 0.12,  误差 0.005) ✅
  2  × 0.01733 + (-0.11) = -0.0753 (原 -0.08, 误差 0.005) ✅
  15 × 0.01733 + (-0.11) = 0.1500  (原 0.15,  误差 0)     ✅
  0  × 0.01733 + (-0.11) = -0.1100 (原 -0.11, 误差 0)     ✅

子块 2 反量化 (精度的关键！):
  11 × 0.00733 + (-0.03) = 0.0506   (原 0.05,  误差 0.0006) ✅
  0  × 0.00733 + (-0.03) = -0.0300  (原 -0.03, 误差 0)      ✅
  15 × 0.00733 + (-0.03) = 0.0800   (原 0.08,  误差 0)      ✅
  1  × 0.00733 + (-0.03) = -0.0227  (原 -0.02, 误差 0.003)  ✅

  → 子块 2 的所有小数都被精确保留了
```

**对比 Q4_0 会发生什么？**

Q4_0 用全局 scale，所有子块共用。以整个 12 个数的最大范围算：

```
Q4_0 全局 scale = 1.00 / 15 ≈ 0.0667 (由子块 1 的范围决定)

用这个 scale 量化子块 2:
  0.05  → round(0.05 / 0.0667)  = 1     反量化: 1 × 0.0667 = 0.067  (误差 0.017)  ❌
  -0.03 → round(-0.03 / 0.0667) = 0     反量化: 0 × 0.0667 = 0      (误差 0.03)   ❌❌
  0.08  → round(0.08 / 0.0667)  = 1     反量化: 1 × 0.0667 = 0.067  (误差 0.013)  ❌
  -0.02 → round(-0.02 / 0.0667) = 0     反量化: 0 × 0.0667 = 0      (误差 0.02)   ❌❌

→ 子块 2 的 4 个数，2 个被"吃成了 0"
→ 信息严重丢失！
```

**精度对比汇总**：

| 场景 | 子块 2 的最大误差 |
|---|---|
| Q4_0（全局 scale） | **0.03** |
| Q4_K（子块独立 scale） | **0.003** |
| **精度提升** | **约 10 倍** |

**Step 5：完整存储（简化版本）**

```
Q4_K 存这 12 个权重的结构:
  
  超级块头:
    global_scale (FP16) = 0.06667 (取最大的 sub_scale)  → 2 字节
    global_min   (FP16) = -0.45   (取最小的 min)        → 2 字节
  
  子块 0:
    sub_scale_0 (6-bit) = 3/63 × 63 ≈ 3   (0.01733/0.06667 × 63)  ← 0.75 字节
    sub_min_0   (6-bit) = ...                                      ← 0.75 字节
    权重值: [13, 2, 15, 0]  → 4 × 4-bit = 2 字节
  
  子块 1:
    sub_scale_1 (6-bit) = 63   (0.06667/0.06667 × 63)             ← 0.75 字节
    sub_min_1   (6-bit) = ...                                      ← 0.75 字节
    权重值: [13, 2, 15, 0]  → 2 字节
  
  子块 2:
    sub_scale_2 (6-bit) = 7    (0.00733/0.06667 × 63)             ← 0.75 字节
    sub_min_2   (6-bit) = ...                                      ← 0.75 字节
    权重值: [11, 0, 15, 1]  → 2 字节
  
总计: 4 + 3 × (0.75 + 0.75 + 2) = 4 + 10.5 = 14.5 字节
```

**对比 Q4_0 存储**：

```
Q4_0 存这 12 个权重:
  global_scale (FP16)  → 2 字节
  12 × 4-bit 权重     → 6 字节
  总计: 8 字节

Q4_K 多用: 14.5 - 8 = 6.5 字节
增加了约 80%（玩具例子数据少，真实 256 个权重只多 12.5%）

换来的收益: 精度提升约 10 倍
```

**Swift 心智模型**

```swift
// 用 Swift 代码完整实现 Q4_K 量化
struct Q4_K_SuperBlock {
    let globalScale: Float      // FP16，再缩放因子
    let globalMin: Float        // FP16
    let subScales: [UInt8]      // 8 个 6-bit sub_scale（这里用 UInt8 简化）
    let subMins: [UInt8]        // 8 个 6-bit sub_min
    let weights: [UInt8]        // 256 个 4-bit 权重
}

func quantizeToQ4_K(fp16: [Float]) -> Q4_K_SuperBlock {
    assert(fp16.count == 256, "一个超级块必须是 256 个权重")
    
    // Step 1: 分成 8 个子块
    var subBlocks: [[Float]] = []
    for i in 0..<8 {
        subBlocks.append(Array(fp16[i*32..<(i+1)*32]))
    }
    
    // Step 2: 每个子块独立计算 scale 和 min
    var subScales: [Float] = []
    var subMins: [Float] = []
    for block in subBlocks {
        let min = block.min()!
        let max = block.max()!
        subScales.append((max - min) / 15.0)
        subMins.append(min)
    }
    
    // Step 3: 全局 scale（用于压缩 sub_scale 到 6-bit）
    let globalScale = subScales.max()!
    let globalMin = subMins.min()!
    
    // Step 4: 压缩每个子块的 scale 到 6-bit
    let compressedScales = subScales.map { UInt8($0 / globalScale * 63) }
    let compressedMins = subMins.map { UInt8(($0 - globalMin) / (globalScale * 63) * 63) }
    
    // Step 5: 量化每个权重到 4-bit
    var quantizedWeights: [UInt8] = []
    for (i, block) in subBlocks.enumerated() {
        let scale = subScales[i]
        let min = subMins[i]
        for w in block {
            let q = UInt8(round((w - min) / scale))
            quantizedWeights.append(Swift.min(q, 15))
        }
    }
    
    return Q4_K_SuperBlock(
        globalScale: globalScale,
        globalMin: globalMin,
        subScales: compressedScales,
        subMins: compressedMins,
        weights: quantizedWeights
    )
}

// 推理时反量化
func dequantizeQ4_K(_ block: Q4_K_SuperBlock) -> [Float] {
    var result: [Float] = []
    for i in 0..<8 {
        // 恢复子块的 scale 和 min
        let subScale = block.globalScale * Float(block.subScales[i]) / 63.0
        let subMin = block.globalMin + block.globalScale * 63.0 * Float(block.subMins[i]) / 63.0
        
        for j in 0..<32 {
            let q = block.weights[i * 32 + j]
            result.append(Float(q) * subScale + subMin)
        }
    }
    return result
}
```

```
Q4_0 (整块共享 scale):
       所有子块共用同一套"量化档位"
       
       档位 (16 个): ●  ●  ●  ●  ●  ●  ●  ●  ●  ●  ●  ●  ●  ●  ●  ●
      
       子块 0 (范围小):  ▲ ▲  ▲  ▲                   ← 只用前几个档位
       子块 1 (范围大):  ▲ ▲ ▲ ▲ ▲ ▲ ▲ ▲ ▲ ▲ ▲ ▲     ← 档位充分利用
       子块 2 (范围小):  ▲                           ← 只能落在第 1 个档位! 信息严重丢失

       子块 1 档位: ●●●●●●●●●●●●●●●●      专为 [-0.45, 0.55] 设计
```

### 8.6 不同量化的权衡

## 九、GGUF 文件到底装了什么？

读到这里你应该对 Transformer 的所有组件都有了理解。现在看一下**这些组件对应的数据是怎么存在 GGUF 文件里**，以及**推理时怎么用**。

先回答一个新手最容易误解的问题：

> **问：GGUF 里存的 Q、K、V 张量，是每次用户输入时用的"Q/K/V 向量"吗？**

**答：不是！** GGUF 里存的是"**生成 Q/K/V 向量的权重矩阵**"（W_q, W_k, W_v），不是 Q/K/V 向量本身。这两个概念必须分清。

### 9.1 核心澄清：权重矩阵 vs 向量

| 概念 | 是什么 | 存在哪里 |
|---|---|---|
| **权重矩阵** `W_q`/`W_k`/`W_v` | 训练得到的"投影矩阵"，固定不变 | ⭐ **GGUF 文件里** |
| **向量** `Q`/`K`/`V` | 运行时用权重矩阵乘以输入算出来的 | **临时计算，用完丢或进 KV Cache** |

#### 工厂类比：模具 vs 产品

```
W_q, W_k, W_v 权重矩阵  = 工厂的"模具"（训练时铸造好，固定不变）
                          每次推理都用同一套模具

Q, K, V 向量           = 用模具生产出的"产品"
                          每个 token 每次推理都不一样
                          产品是临时的（或进 KV Cache 短暂保存）

GGUF 文件 = 存放模具的仓库
```

#### 代码层面对比

```swift
// 从 GGUF 一次性加载的"模具"（整个 App 生命周期都在内存里）
let W_q: [[Float]]  // [1536, 1536]  从 blk.0.attn_q.weight 加载
let W_k: [[Float]]  // [1536, 256]   从 blk.0.attn_k.weight 加载
let W_v: [[Float]]  // [1536, 256]   从 blk.0.attn_v.weight 加载

// 每个 token 每次推理都重新算的"产品"
func selfAttention(tokenEmbedding: [Float]) {
    let Q = matmul(tokenEmbedding, W_q)  // ← 每次都算，Q 是产品
    let K = matmul(tokenEmbedding, W_k)
    let V = matmul(tokenEmbedding, W_v)
    // ... 用 Q/K/V 做 attention
}
```

```
output_norm.weight          [1536]         ← 最后一层归一化
output.weight               [151936, 1536] ← LM Head (把向量映射到词表)
  (可选择和 embedding 共享权重，节省存储)
```

**完整计数**：

```
张量总数: 1 (embedding) + 28 × 10 (每层) + 2 (output) ≈ 283 个张量
参数总量: ~1.5B (对得上 Qwen2.5-1.5B 的名字)
量化后大小: ~1.0 GB (Q4_K_M)
```

### 9.3 FFN 里的 gate / up / down 到底是什么？

这是客户端同学的第二个高频疑问。为什么 FFN 有三个矩阵？传统神经网络不是只有"线性层 + 激活函数"吗？

#### 传统 FFN vs SwiGLU FFN

```
传统 FFN (ReLU/GELU，已过时):
  input ──▶ W_up ──▶ [扩展到 8960 维] ──▶ ReLU ──▶ W_down ──▶ output
  
  只有 2 个矩阵: W_up, W_down
  激活函数: 硬阈值（>0 留，<0 丢）

SwiGLU FFN (现代主流):
                    ┌──▶ W_up  ──▶ [扩展到 8960 维]
                    │                  │
  input ────────────┤                  │
                    │                  ▼
                    └──▶ W_gate ──▶ SiLU(...) ──▶ (× 按元素相乘)
                                                    │
                                                    ▼
                                                  W_down ──▶ output
  
  三个矩阵: W_gate, W_up, W_down
  激活函数: 软门控（0-1 连续调整）
```

#### 三个矩阵的分工

```
W_up   (1536 → 8960):  "内容提取器"
  把输入投影到一个更大的空间
  提取 8960 种"特征内容"

W_gate (1536 → 8960):  "门控器 / 阀门"
  也投影到 8960 维
  经过 SiLU 激活，变成 0-1 之间的"门控信号"
  决定"W_up 的每个维度要放行多少"

W_down (8960 → 1536):  "压缩器"
  把门控后的结果压回 1536 维
  完成一次"思考"
```

#### 直观类比：带阀门的水管

```
想象:
  W_up 是一根水管，有 8960 个出水口
  W_gate 是每个出水口的阀门
  W_down 是下游的汇流管

input 向量流进来:
  │
  ├─▶ W_up: 把水分成 8960 股，每股代表一种"特征内容"
  │    │
  │    股 1: 0.9 (特征 1 的强度)
  │    股 2: 0.3
  │    股 3: 0.7
  │    ...
  │    股 8960: 0.1
  │
  └─▶ W_gate: 算出每个阀门的开度
       │
       阀门 1: 开度 0.95 (让特征 1 几乎全通过)
       阀门 2: 开度 0.1  (挡住特征 2)
       阀门 3: 开度 0.0  (完全关闭)
       ...
       阀门 8960: 开度 0.5
  
按元素相乘 (filtered = W_up × W_gate):
  过滤后的流量 1: 0.9 × 0.95 = 0.855  ← 保留
  过滤后的流量 2: 0.3 × 0.1  = 0.030  ← 基本被挡
  过滤后的流量 3: 0.7 × 0.0  = 0.000  ← 完全被挡
  ...

W_down: 把保留下来的特征汇流成最终输出 (1536 维)
```

#### 为什么门控机制更强？

核心优势：SwiGLU 提供了"**乘法**"这个关键操作，让模型能学习**条件逻辑**：

```
"如果特征 A 被激活，那么让特征 B 通过"

这是 ReLU 做不到的:
  ReLU 只能做"线性切断"
  不能做"根据条件 A 控制 B"

SwiGLU 通过 gate × up 的乘法实现了这个
```

#### 完整 Swift 代码

```swift
func ffnForward(input: [Float],
                W_gate: [[Float]],   // blk.N.ffn_gate.weight
                W_up: [[Float]],     // blk.N.ffn_up.weight
                W_down: [[Float]]) -> [Float] {  // blk.N.ffn_down.weight
    
    // Step 1: 两路并行投影到 8960 维
    let upValues = matmul(input, W_up)       // [1, 8960] 内容
    let gateValues = matmul(input, W_gate)   // [1, 8960] 门控信号
    
    // Step 2: 门控值经过 SiLU 激活 (变成 0-1 阀门开度)
    var gateActivated = [Float](repeating: 0, count: 8960)
    for i in 0..<8960 {
        // SiLU(x) = x / (1 + exp(-x))
        gateActivated[i] = gateValues[i] / (1 + exp(-gateValues[i]))
    }
    
    // Step 3: 按元素相乘 = 用阀门筛选内容
    var filtered = [Float](repeating: 0, count: 8960)
    for i in 0..<8960 {
        filtered[i] = gateActivated[i] * upValues[i]
    }
    
    // Step 4: 压回 1536 维 = 融合特征
    let output = matmul(filtered, W_down)  // [1, 1536]
    
    return output
}
```

#### 对比传统 FFN

| | 传统 ReLU FFN | SwiGLU FFN |
|---|---|---|
| 矩阵数量 | 2 (W_up, W_down) | 3 (W_gate, W_up, W_down) |
| 参数量 | 基准 | 1.5× |
| 激活方式 | 硬阈值（>0 留，<0 丢） | 软门控（0-1 连续） |
| 表达能力 | 弱 | 强（有乘法关系） |
| 现代使用 | 已淘汰 | Llama、Qwen、Mistral 全部用 |

### 9.5 运行时：每一层都用到哪些张量

这是核心问题的直观答案——**用户输入一句话，GGUF 里的每个张量都会被"过"一遍**：

```
用户输入: "你好"
           │
           ▼
┌──────────────────────────────────────────────┐
│  Step 1: Tokenize                              │
│  "你好" → token IDs [45678]                     │
└────────────────────┬─────────────────────────┘
                     ▼
┌──────────────────────────────────────────────┐
│  Step 2: Embedding 查表                       │
│  使用: token_embd.weight                       │
│                                                │
│  token 45678 → 查第 45678 行                   │
│  → 得到 1536 维初始向量                         │
└────────────────────┬─────────────────────────┘
                     ▼
┌──────────────────────────────────────────────┐
│  Step 3-30: 28 层 Transformer Block            │
│                                                │
│  每一层用到 blk.N.* 的 10 个张量:               │
│                                                │
│  ── Attention 部分 ──                          │
│  1. rmsNorm(x, blk.N.attn_norm.weight)         │
│  2. Q = matmul(x, blk.N.attn_q.weight)         │
│  3. K = matmul(x, blk.N.attn_k.weight)         │
│  4. V = matmul(x, blk.N.attn_v.weight)         │
│  5. 算 attention (Q·K, softmax, ·V)            │
│  6. x = matmul(attn, blk.N.attn_output.weight) │
│  7. 残差: x = x + input                         │
│                                                │
│  ── FFN 部分 ──                                │
│  8.  rmsNorm(x, blk.N.ffn_norm.weight)         │
│  9.  up = matmul(x, blk.N.ffn_up.weight)       │
│  10. gate = SiLU(matmul(x, blk.N.ffn_gate.weight)) │
│  11. x = gate ⊙ up  (按元素相乘)               │
│  12. x = matmul(x, blk.N.ffn_down.weight)      │
│  13. 残差: x = x + input                        │
│                                                │
│  → 28 层共 28 × 10 = 280 次张量使用             │
└────────────────────┬─────────────────────────┘
                     ▼
┌──────────────────────────────────────────────┐
│  Step 31: 最后归一化 + LM Head                 │
│  使用: output_norm.weight, output.weight       │
│                                                │
│  → 得到 151936 维 logits                        │
│  → 采样得到下一个 token                         │
└────────────────────┬─────────────────────────┘
                     ▼
                 下一个 token
```

**关键认知**：

```
每生成 1 个 token，需要:
  - 使用 GGUF 里所有 283 个张量
  - 进行几十亿次浮点运算
  
生成 100 个 token 的回答 ≈ 把整个 GGUF "读" 101 次

这就是为什么"内存带宽"是推理瓶颈（参考 01 文档）
——不是算力不够，是数据搬运太慢
```

### 9.6 关键观察

1. **元数据是自描述的**：拿到 GGUF 文件，不用额外配置就能推理——所有必要信息（架构、维度、tokenizer、chat template）都在文件里。

2. **Chat template 也存在文件里**：`tokenizer.chat_template` 字段包含完整的 Jinja2 模板，llama.cpp 会自动用它。

3. **不同层用不同量化**：attention 用 Q4_K 省空间，FFN_down 和 LM Head 用 Q6_K 保精度——这就是 Q4_K_M 里 "M" 的含义（Medium，重要层用更高精度）。

4. **归一化层不量化**：`*_norm.weight` 都是 F32 全精度，因为它们参数很少（只有 1536 个值）但对结果敏感，量化得不偿失。

5. **一些层会"共享"权重**：有些模型（如 Gemma）让 `output.weight` 和 `token_embd.weight` 共享同一个矩阵，节省 ~15% 参数。GGUF 用 `tie_word_embeddings` 元数据标记。

### 9.8 一句话总结

> **GGUF 里存的全是"模具"——训练好的权重矩阵，永远固定不变。**

>
> **所以 GGUF 文件是模型的"骨架"，装满了可以被复用的权重矩阵。**

### 10.2 实际推理

```swift
extension LlamaEngine {
    func generate(prompt: String) -> AsyncStream<String> {
        AsyncStream { continuation in
            Task {
                // 1. Tokenize
                let tokens = tokenize(prompt)
                
                // 2. Prefill
                prefill(tokens: tokens)
                
                // 3. Decode 循环
                var n_cur = tokens.count
                while n_cur < 4096 {
                    let nextToken = sample()
                    
                    if nextToken == llama_token_eos(model) {
                        break  // 遇到结束符
                    }
                    
                    let piece = detokenize(nextToken)
                    continuation.yield(piece)
                    
                    // 把新 token 喂回模型
                    decode(token: nextToken, position: n_cur)
                    n_cur += 1
                }
                
                continuation.finish()
            }
        }
    }
}
```

## 十二、这一篇的核心收获

1. **Transformer = 28 层相同结构的叠加**，每层做两件事：注意力 + FFN
2. **Self-Attention 的本质**：每个 token 根据 Q/K 相似度加权融合其他 token 的 V
3. **因果掩码**让 LLM 只看过去，是生成模型的关键约束
4. **GQA** 是端侧模型的关键优化，大幅降低 KV Cache 开销
5. **FFN 占模型大部分参数**，它是非线性来源 + 知识存储库，这就是 MoE 拆 FFN 的原因
6. **残差连接**解决梯度消失，让深层 Transformer（几十上百层）得以训练
7. **Chat Template 不对就一切白搭**，必须严格遵循模型训练时的格式
8. **Q4_K_M 是 iPhone 端的甜点量化**，大小和质量最优平衡
9. **GGUF 是自描述格式**，元数据全在文件里，拿来就能用
